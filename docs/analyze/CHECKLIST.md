# ChatBoxApp 代码质量审查清单

> 创建时间: 2026-02-01
> 目的: 系统性检查代码库，识别潜在的"屎山"风险点
> 原则: 只分析，不修改

---

## 审查维度

### 1. 架构一致性 (Architectural Consistency)

- [ ] **1.1 依赖方向**：检查是否存在反向依赖（如 services → pages）
- [ ] **1.2 层级边界**：UI 层是否直接操作 Hive/数据库
- [ ] **1.3 全局状态**：全局变量使用是否合理，是否存在隐式耦合
- [ ] **1.4 模块职责**：单一职责原则遵守情况

### 2. 代码复杂度 (Code Complexity)

- [ ] **2.1 文件行数**：超过 500 行的文件
- [ ] **2.2 函数长度**：超过 50 行的函数
- [ ] **2.3 嵌套深度**：超过 4 层嵌套的代码块
- [ ] **2.4 圈复杂度**：条件分支过多的函数

### 3. 代码重复 (Code Duplication)

- [ ] **3.1 逻辑重复**：相似功能在多处实现
- [ ] **3.2 模式重复**：可抽象为通用工具的重复模式
- [ ] **3.3 常量重复**：魔法数字、硬编码字符串

### 4. 错误处理 (Error Handling)

- [ ] **4.1 异常吞没**：空 catch 块或只打印不处理
- [ ] **4.2 错误传播**：异常是否正确向上传播
- [ ] **4.3 边界检查**：null 检查、空集合检查
- [ ] **4.4 资源释放**：dispose、close 是否正确调用

### 5. 类型安全 (Type Safety)

- [ ] **5.1 dynamic 使用**：不必要的 dynamic 类型
- [ ] **5.2 类型转换**：不安全的 as 转换
- [ ] **5.3 null 安全**：可空类型处理是否充分

### 6. API 设计 (API Design)

- [ ] **6.1 接口一致性**：相似功能的 API 风格是否统一
- [ ] **6.2 参数设计**：参数数量、可选参数合理性
- [ ] **6.3 返回值设计**：void vs Future、单值 vs 集合

### 7. 状态管理 (State Management)

- [ ] **7.1 状态分散**：同一状态在多处维护
- [ ] **7.2 状态同步**：不同组件间状态是否同步
- [ ] **7.3 状态持久化**：持久化与内存状态的一致性

### 8. 并发安全 (Concurrency Safety)

- [ ] **8.1 竞态条件**：异步操作的时序问题
- [ ] **8.2 资源竞争**：共享资源的并发访问
- [ ] **8.3 取消处理**：异步操作的取消逻辑

### 9. 可测试性 (Testability)

- [ ] **9.1 依赖注入**：硬编码依赖 vs 可注入
- [ ] **9.2 纯函数比例**：副作用函数占比
- [ ] **9.3 mock 友好度**：是否易于 mock

### 10. 文档与注释 (Documentation)

- [ ] **10.1 公共 API 文档**：公共方法是否有 dartdoc
- [ ] **10.2 复杂逻辑注释**：非显而易见的逻辑是否有解释
- [ ] **10.3 过时注释**：注释与代码是否一致

### 11. 技术债务标记 (Tech Debt Markers)

- [ ] **11.1 TODO/FIXME**：未完成项统计
- [ ] **11.2 临时方案**：标记为"临时"的代码
- [ ] **11.3 废弃代码**：未使用的代码

### 12. 安全性 (Security)

- [ ] **12.1 敏感信息**：API Key、密码硬编码
- [ ] **12.2 输入验证**：用户输入的校验
- [ ] **12.3 数据暴露**：日志中是否泄露敏感数据

## 审查原则

1. **客观性**：基于代码事实，非主观偏好
2. **可操作性**：问题描述具体，可定位
3. **优先级明确**：区分严重/警告/建议
4. **不改代码**：只分析记录，改动另行安排
