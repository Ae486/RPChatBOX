# 代码审查要求（商业化产品视角 / Flutter 多端客户端）

## 0. 使用方式（强制）
请以“商业化产品技术负责人 + 安全/隐私负责人”的视角执行审查，并满足：
- 每条问题必须给出**证据**：`文件路径:行号` + 关键代码/配置点（可简述，避免贴大段）
- 每条问题必须给出**结论可执行**的修复建议：修复点、建议方案、回归验证方式、预估工作量（S/M/L）
- 先做“上线阻断项（P0）”筛查，再做“高优先级（P1）”，最后再做优化项（P2/P3）

## 1. 审查输入（审查前请确认）
- 审查对象：仓库分支/commit、构建版本（`pubspec.yaml` 的 `version`）
- 目标平台：Android / iOS / Windows / macOS / Linux / Web（至少声明主发平台）
- 数据分级：是否包含 API Key、聊天内容、附件、用户标识等（属于敏感数据/个人数据）
- 运行方式：本地直连第三方 LLM？是否存在企业代理/聚合网关？（影响合规与安全边界）

## 2. 风险分级（统一口径）
- **P0（上线阻断）**：核心功能不可用、发布配置错误、安全/隐私重大问题、数据丢失/不可逆、可被远程利用的高危漏洞
- **P1（高风险）**：高频崩溃/卡死、敏感信息可能泄露、严重性能问题、跨平台主路径不一致导致大面积故障
- **P2（中风险）**：可维护性/扩展性明显不足、边界场景缺陷、质量工程缺口（测试/监控）导致回归风险高
- **P3（建议优化）**：代码风格、轻量重构、体验提升、长期技术债清理

> 每条问题请同时给出：**风险等级（P0~P3）**、**影响范围**、**触发条件/复现步骤**、**建议修复**、**验证方案**。

## 3. 审查范围（本项目建议优先覆盖）
### 3.1 核心业务链路
- 主页面与会话流转：`lib/pages/*`
- 聊天视图与流式渲染：`lib/widgets/*`、`lib/controllers/*`
- Provider/网络请求/流式解析：`lib/adapters/*`、`lib/services/dio_service.dart`
- 数据持久化与迁移：`lib/services/*`、`lib/models/*`（Hive/SharedPreferences）

### 3.2 高风险组件（安全/合规/稳定性）
- WebView/模板资产：`lib/widgets/mermaid_renderer.dart`、`assets/web/*`
- 文件/附件解析：`lib/services/file_content_service.dart`（PDF/DOCX/压缩包等）
- 平台工程配置：`android/`、`ios/`、`macos/`、`windows/`、`web/`
- 测试与质量体系：`test/`、`analysis_options.yaml`

## 4. 审查维度（Checklist）

### 4.1 商业化与发布就绪
- Android/iOS/桌面端是否具备发布所需的最小配置（权限、签名、版本号、隐私声明、崩溃收敛）
- 关键依赖的**许可证/商业授权**是否明确（尤其是 PDF/Office 解析类、UI 组件类）
- 是否存在会导致商店审核失败的风险（例如 WebView 动态加载远程脚本/远程代码执行）

### 4.2 安全与隐私（客户端重点）
- **敏感数据存储**：API Key/Token、聊天内容、附件摘要等是否明文落盘？是否使用 Keychain/Keystore/加密存储？
- **日志脱敏**：请求头/URL/响应体中是否包含敏感字段？Debug 日志是否可能被收集并外泄？
- **WebView 安全**：JavaScript 开启策略、是否加载远程资源、是否允许任意导航、模板注入风险
- **外部链接与协议**：`url_launcher` 是否限制 scheme/是否有二次确认（防钓鱼/恶意跳转）
- **供应链安全**：依赖版本是否锁定、是否存在明显过旧或弃用依赖；是否有 SCA/漏洞扫描流程

### 4.3 稳定性与鲁棒性
- 流式请求是否支持**真正取消**（网络层 cancel + UI 状态收敛 + 资源释放）
- 超时、重试、降级策略是否一致且可控（尤其是流式 SSE/长连接）
- 异常是否被统一包装为可展示/可定位的信息（用户提示 + 可观测日志）

### 4.4 性能与资源
- UI 渲染是否避免 O(n^2) 拼接、过度 setState、长列表/长会话内存膨胀
- 文件处理/解码是否在主线程做重活（同步 IO/解压/大文件 base64）
- 缓存与清理：图片缓存/Hive 数据增长是否可控，是否提供清理入口与上限

### 4.5 代码结构与可维护性
- 模块边界是否符合项目约定（UI → controllers → services → adapters → models/storage）
- 配置体系是否单一且一致（避免多套设置并存导致行为不确定）
- 关键路径是否有最小保护网（单元/Widget/Golden 测试）与回归脚本

### 4.6 可观测性（产品级）
- 是否具备 crash 上报/错误聚合的接入点（可选但商业化建议具备）
- 日志是否分级、可开关、默认不泄露敏感信息；是否有用户隐私开关/同意机制
