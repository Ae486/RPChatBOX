# 容易忽略的点与开放问题（Draft）

> 目标：把“很容易到开发中期才发现的大坑”提前写出来，作为讨论清单与后续验收依据。
>
> 本文不做工程落地映射，只描述需求/规则/边界。
>
> 最后更新：2026-01-07

---

## 0. 你已经做对了哪些关键选择（避免走弯路）

- 模块可插拔 + 禁用即彻底忽略（不注入、不维护、不可编辑）
- Canon 只写明确事实 + 人类在环确认 + 可回滚
- Key Events（全局）与 Foreshadow 关联层（Linker）分离
- 伏笔 Tracker 双轴（证据积累/回收接近）+ 证据链 + 可手动 delta
- Standard/Immersive 模式（权限与沉浸兼顾）

下面是“仍然容易被忽略”的部分。

---

## 1. 事实的来源可信度（Epistemic Model）

同一句话可能来自不同来源，可信度不同：

- 用户陈述（通常可信，但也可能是“角色撒谎/误导/记错”）
- 角色台词（更可能包含谎言/误解）
- 旁白/叙述（通常更接近“客观事实”）
- 系统工具输出（通常可信，但也可能过期或不完整）

建议在 Canon/Draft 之外，再补一个最小的“来源标签”：
- `source_type = user | narrator | character_speech | system | tool`

开放问题：
1) 当用户以“角色身份”说的话，默认算不算客观事实？还是默认进入“角色主观记忆”？
2) 你是否需要支持“角色撒谎/误导”的显式标记？（否则系统可能把谎言写入 Canon）

✅ 已确认补充：
- 用户以“角色身份”输入通常是为了引导剧情推进，系统按“正常用户输入”处理（但不自动写入 Canon；Canon 仍需确认）。
- ✅ 已确认：可以增加最小 `source_type`（你希望 agent 能在条目中表达“谁撒谎、为了什么”，`source_type` 作为结构化辅助信号，降低误写入 Canon 的风险）。

---

## 2. 分支/回滚与后台维护的竞态

你允许后台维护、并且支持回滚；但要注意竞态：

- 用户切换分支（active chain 变了）
- Orchestrator 在后台跑了提议/关联/总结
- 用户回滚某次更新

如果没有“作用域与版本”约束，很容易出现：回滚后又被后台任务写回去。

开放问题：
1) 后台任务的作用域默认是否严格绑定 active chain（分支）？
2) 回滚后是否需要“阻止旧任务写入”的语义（例如：过期任务自动丢弃）？

---

## 3. 角色信息不对称（Secrets）

角色扮演里常见：玩家知道但角色不知道；或某角色有秘密。

你已经允许“角色误解/错误信念”，这是关键的一步；建议再明确一条：
- **注入时的“视角过滤”**：写给某角色视角的回复，不应注入该角色未知的信息。

开放问题：
1) 你希望默认视角是“用户角色第一视角”，还是“旁白/导演视角”？还是由 Scene/Style 决定？
2) 是否需要一个可选模块/视图：`Secrets`（仅系统可见，避免泄露）？

✅ 已确认补充：
- 当某角色发言/该角色视角输出时，上下文以该角色的“角色卡 + Delta + 角色记忆/已知信息”为准（避免注入该角色未知的信息）。

---

## 4. Proposal 过载与“打断沉浸”的控制

当系统变聪明后，提议会变多：关键事件提议、伏笔关联提议、状态变更提议、冲突提议……

如果每条都弹窗，会毁掉体验。

建议明确：
- 哪类提议必须即时弹窗（高风险、会改 Canon 的）
- 哪类提议进入“待处理队列”（低风险、可批量）
- Immersive 模式下默认只弹“必须弹”的

开放问题：
1) 你是否接受“批量处理”提议？（例如一次性确认/拒绝 5 条）
2) 你希望“待处理队列”的入口放在哪里更不破坏沉浸？

补充：
- Proposal 类型清单已单独整理：`docs/roleplay-feature/15-PROPOSALS-CATALOG.md`。

---

## 5. 一致性闸门（Consistency Gate）覆盖范围

你最关心的低级错误（外观/状态/世界线矛盾），闸门是非常高 ROI 的防线。

开放问题：
1) 你更希望闸门是“永远开启”还是“只在风险高时开启”（例如长对话、关键场景、生成后自检失败）？
2) 闸门发现错误时，是“自动重写回复”还是“提示用户选择纠错/剧情变化”？

---

## 6. 导出 JSON 的隐私与脱敏

你希望创意工坊式导出，但 JSON 里可能包含私人信息（用户名、聊天内容、敏感设定）。

开放问题：
1) 导出是否需要“脱敏选项”？（例如删除对话原文、只保留条目摘要与结构）
2) 是否需要“只导出模块配置，不导出记忆数据”的轻量分享模式？

✅ 已确认补充：
- 你当前导出目标主要是 style/角色卡/世界书等不含隐私内容；对“误导出隐私”的风险暂不做强约束（更多视为用户自担）。

---

## 7. 质量评估：怎么证明“确实变稳了”

如果没有验收指标，后期很难判断改动是否有效。

建议最少定义 3 类指标：
- 一致性错误率：外观/状态/世界线矛盾的频率
- 记忆命中率：用户提到旧伏笔/旧设定时，系统召回正确条目的比例
- token 成本：每轮注入长度与被裁剪原因统计

开放问题：
1) 你是否愿意维护一组“回归测试剧本”（几段固定对话）用于验证一致性？

---

## 8. 下一步最值得讨论的 5 个问题（建议按顺序）

1) Proposal 类型与处理策略：哪些必弹窗、哪些队列、哪些可静默？
2) 事实来源标签：是否需要 `source_type`（user/narrator/character_speech/tool）？
3) 默认视角与 Style：未配置 style 时不强约束；配置后按 style 约束（你已确认）。
4) Key Event 判定最小规则：哪些变化必然是关键？（用于提升系统判定的可解释性）
5) 导出脱敏：是否需要、需要到什么程度？
