# Key Events / Linker / 一致性闸门（Draft）

> 目标：把“关键事件驱动的伏笔追踪（含进度条）”做成可解释、可回滚、可控 token 的系统能力；并补上一道“生成后一致性检查闸门”，降低长对话下的低级设定错误（外观/状态/世界线）。
>
> 本文为需求/设计说明，不做工程落地映射。
>
> 最后更新：2026-01-07

---

## 0. 核心结论（本文件解决什么）

你已经确认了三件关键决策：

1) **关键事件（Key Event）系统判定为主**，用户可手动编辑/标记。  
2) **伏笔到期单位按 N 个关键事件**（从埋下后累计）。  
3) 伏笔 Tracker 同时有两条轴：  
   - **证据积累**（evidence accumulation）  
   - **回收接近**（payoff proximity）  
   二者按权重综合，权重可调整。

本文件把它们串成一套“可执行的概念流程”，并补充一个长期很关键但容易忽略的能力：**一致性闸门**。

---

## 1. Key Event（关键事件）到底是什么？

### 1.1 定义（与伏笔无关）

- Key Event 是 Timeline/Event Ledger 里的“高价值客观事件条目”。  
- Key Event **不要求**与任何伏笔相关；它的作用是：
  - 表示“世界线推进了一个重要节点”（用于到期单位）
  - 作为证据源（Proposal/进度条/冲突判断引用）

### 1.2 “与伏笔相关”发生在 Linker 层

为了避免混淆，需要明确两层：

- **Key Event（全局）**：发生了什么。  
- **Foreshadow‑Linked Event（关联层）**：这个事件对某条伏笔是正相关/负相关/无关，以及影响多少。

---

## 2. Key Event 的最小判定标准（建议先定 v0 规则）

你希望系统自动判定“是否关键”，但又必须可解释。建议 v0 先定义一个可审计的“最小集合”：

### 2.1 事件类型（建议至少覆盖）

- **角色状态变化**：伤势、能力、阵营/立场、关系状态显著变化
- **物品/资源变化**：关键道具获得/失去/损坏/消耗
- **地点/场景转移**：进入关键地点、离开关键地点、封锁/解锁通路
- **信息揭示/反转**：身份揭示、真相揭示、重大误解澄清
- **冲突阶段变化**：从潜伏→对峙、从对峙→爆发、从爆发→收束
- **世界规则变更**：新增/修改世界设定（需人类在环确认）

### 2.2 系统输出必须带“理由”

系统判定某条为 Key Event 时，至少要记录：
- 为什么是关键（归类到哪个类型）
- 引用证据（消息片段或已存在事件）
- 重要性/置信度（Draft）

### 2.3 用户手动编辑

用户可对 Key Event：
- 标记为关键/非关键
- 编辑事件内容（会留下版本记录）
- 调整重要性（可选）

> 注意：禁用相关模块时不可编辑（你已确认）。

## 3. Linker（关联器）：把事件挂到伏笔下面

### 3.1 关联条目：ForeshadowEventLink（建议作为 Entry）

Linker 产出“关联条目”，字段建议见 `11` 文档；这里强调两点：

1) `relation` 支持 `positive/negative/neutral`  
2) `delta_evidence/delta_payoff` 允许正负，并支持用户手动调整（影响率/影响量）

### 3.2 正相关 vs 负相关：怎么理解“负相关”

你问“要不要包含负相关”。建议：**包含，但要更谨慎**。

- 负相关的典型意义：
  - **阻碍**：短期更难回收（例如关键道具损坏）
  - **反证**：削弱某条 Draft 推测（不是改写客观事实）
  - **误导**：烟雾弹（写作常用；可能让回收接近下降但证据积累上升）

所有关联与 delta 都应默认 Draft，并带证据链与理由，避免“模型拍脑袋降进度”。

你要求“复杂但要可控”，这里给出明确默认策略：

- ✅ 允许存在 `negative`（作为阻碍/反证/误导的叙事素材）
- ✅ 默认 Linker **不自动生成 negative**：自动关联只生成 `positive/neutral`
- ✅ 若要出现 `negative`：走 Proposal，默认 `review_required`（用户确认或用户手动添加/修改）
- ✅ 真相大白后“negative 变成可用素材/证据”的处理：优先新增 `reinterpretation` 关联（引用回收事件），不静默翻转历史记录

## 4. Tracker 两条轴怎么更新（概念算法）

### 4.1 证据积累（progress_evidence）

来源：**与该伏笔相关的关联条目**（ForeshadowEventLink）

- 每条 Link 提供 `delta_evidence`（可正可负）
- 用户可手动调整某条 Link 的影响量

### 4.2 回收接近（progress_payoff）

来源：**叙事窗口信号**（同样应能落到“事件/状态变化”上）

典型信号：
- 进入回收地点/关键人物同场/关键道具到手（Scene/State 变化，也可写成 Key Event）
- 目标接近完成（Goals 变化，也可写成 Key Event）
- 从埋下后已发生 N 个 Key Event（你选择的到期单位）
- 回收条件被破坏/剧情改道（负向信号）

这些信号可以被 Linker 或专用 agent 产出为 `delta_payoff`，并写入关联条目（或写入“payoff signals”条目），确保可解释与可回滚。

### 4.3 综合进度（progress）

建议：`progress = w_evidence * progress_evidence + w_payoff * progress_payoff`（截断到 0–100）

你已确认：两条轴都需要，并按合适权重综合；权重默认由系统给出，用户可调整。

## 5. “到期提醒（due）”怎么做才不烦人

你提出一个很重要的观点：与其把“到期提醒”做成一个独立机制，不如让它成为 **Tracker/进度条体系的一部分**（更灵活：可以慢慢涨，也可以“自爆式”瞬间拉满）。

因此这里建议把 due 视为 **派生标签/注入选择策略**，而不是一个独立的、刚性的字段或强制提醒：

- “到期”不必单独存 `due=true`：由 `urgency_score` 或 `progress_payoff/progress` 的阈值派生出来即可。
- “N 个关键事件”仍然有价值：它可以作为 **回收接近（progress_payoff）** 的一个信号（“叙事窗口随时间推进逐步打开”）。
- 是否真的回收仍由主对话生成与用户决策共同决定；系统只是用 View 把“最该提醒的那几条”挑出来注入。

同时建议增加一个“减少打断”的策略：
- 低优先级更新进入“待处理 Proposal 队列”
- 在 Scene 切换/用户主动打开“记忆管理页”时再集中处理

## 6. 一致性闸门（Consistency Gate）：我们目前讨论容易忽略但非常关键

你点名的典型错误（发色/瞳色错、关键节点忘、行为语言不符合设定）很多不是“记忆没存”，而是“生成时没对齐约束”。

建议把它作为一个可选闸门（默认低成本触发）：

### 6.1 闸门检查什么（v0 最小集）

- **Character Quick Facts**（外观关键字段、禁忌、口癖、强约束）
- **State/Flags**（伤势/物品/关系状态的硬约束）
- **Timeline Key Events**（不能自相矛盾的已确认节点）

### 6.2 闸门如何处理发现的矛盾（概念）

- 发现矛盾 → 输出“矛盾列表 + 引用证据 + 建议修正方式”
- 修正方式可以是：
  - 让模型“重写一版回复”（不改剧情，只改错误细节）
  - 或弹出提示，用户选择“纠错/剧情变化/忽略”

> 闸门本质是“把错误拦在输出前”，对你最关心的低级错误非常有效。
