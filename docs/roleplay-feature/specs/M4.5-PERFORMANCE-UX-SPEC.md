# M4.5 Performance & UX Optimization Spec

> Worker Isolate 性能优化与用户体验改进规格
>
> 版本: 1.0 | 状态: ✅ Implemented | 日期: 2026-02-01
>
> 实现说明: 超时、取消、进度报告机制已集成到 rp_worker_entry.dart

## 1. 概述

### 1.1 背景

M4 Agent Integration 架构已完成，但存在以下性能和 UX 问题：

| 优先级 | ID | 问题描述 |
|--------|-----|----------|
| P0 | PERF-01 | LLM 代理调用无超时，可永久悬挂 |
| P0 | PERF-02 | 取消操作不可中断 LLM/修复/摘要流程 |
| P0 | UX-01 | 无进度反馈（幽灵加载） |
| P1 | UX-02 | 版本冲突静默失败 |
| P1 | PERF-03 | Worker 冷启动延迟 (最长 10s) |
| P1 | PERF-04 | LLM + JSON 管道叠加延迟 |
| P2 | UX-03 | 错误代码不可读 (E1xx-E5xx) |

### 1.2 目标

- **稳定性**: 消除永久悬挂，确保所有异步操作有超时兜底
- **响应性**: 取消操作 <1s 响应，首包延迟 p95 <3s
- **可感知**: 用户可见 6 阶段进度，错误提示友好化
- **一致性**: 版本冲突有明确提示和恢复选项

## 2. 技术设计

### 2.1 PERF-01: LLM 调用超时机制

**问题**: `_proxyLlmCall` 中 `await completer.future` 无超时，Main Isolate 未回包时永久悬挂。

**方案**:

```dart
const kDefaultLlmTimeoutMs = 30000; // 30 秒默认超时

Future<String> _proxyLlmCall({
  required String systemPrompt,
  required String userPrompt,
  required String modelId,
  int? maxTokens,
  double? temperature,
  int? timeoutMs,
}) async {
  final callId = 'llm_${_llmCallIdCounter++}';
  final completer = Completer<String>();
  _llmCallCompleters[callId] = completer;

  // 发送 LLM 请求
  _sendControl(RpWorkerControl.llmRequest(...));
  _llmCallCount++;

  // 超时 Timer
  final timeout = timeoutMs ?? kDefaultLlmTimeoutMs;
  final timer = Timer(Duration(milliseconds: timeout), () {
    if (!completer.isCompleted) {
      _llmCallCompleters.remove(callId);
      completer.completeError(
        LlmTimeoutException(callId, Duration(milliseconds: timeout)),
      );
    }
  });

  try {
    return await completer.future;
  } finally {
    timer.cancel();
    _llmCallCompleters.remove(callId);
  }
}
```

**配置项**:

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `kDefaultLlmTimeoutMs` | 30000 | 默认 LLM 调用超时 |
| `kLlmRepairTimeoutMs` | 15000 | JSON 修复 LLM 超时（较短） |
| `kSummarizerTimeoutMs` | 20000 | 摘要生成超时 |

### 2.2 PERF-02: 取消信号传播

**问题**: `_cancelRequested` 仅在任务循环间检查，无法中断正在 await 的 LLM 调用。

**方案**: 扩展 `CancellationToken` 支持 `onCancel` 回调。

```dart
// cancellation_token.dart 扩展

class CancellationToken {
  bool _isCancelled = false;
  final _cancelController = StreamController<void>.broadcast();

  bool get isCancelled => _isCancelled;
  Stream<void> get onCancel => _cancelController.stream;

  void throwIfCancelled() {
    if (_isCancelled) throw CancelledException();
  }
}

class CancellationTokenSource {
  final CancellationToken token = CancellationToken();

  void cancel() {
    if (!token._isCancelled) {
      token._isCancelled = true;
      token._cancelController.add(null);
      token._cancelController.close();
    }
  }
}
```

**集成点**:

```dart
// _proxyLlmCall 集成取消
Future<String> _proxyLlmCall(..., {CancellationToken? cancellation}) async {
  cancellation?.throwIfCancelled();

  final callId = 'llm_${_llmCallIdCounter++}';
  final completer = Completer<String>();

  // 监听取消信号
  StreamSubscription? cancelSub;
  if (cancellation != null) {
    cancelSub = cancellation.onCancel.listen((_) {
      if (!completer.isCompleted) {
        _llmCallCompleters.remove(callId);
        completer.completeError(CancelledException());
      }
    });
  }

  try {
    // ... 发送请求和等待响应
    return await completer.future;
  } finally {
    cancelSub?.cancel();
    _llmCallCompleters.remove(callId);
  }
}
```

```dart
        'stageCode': stageCode,

        if (message != null) 'message': message,
```

```dart
    type: RpMessageType.progress,

// 在关键节点调用
Future<void> _handleRequest(...) async {

  final rawOutput = await _proxyLlmCall(...);

  _reportProgress(requestId, RpTaskStage.processing);

  if (parseResult.needsRepair) {

}
```

### 2.4 UX-03: 错误友好化

**问题**: E101-E503 内部错误码对用户无意义。

**方案**: 新增错误映射方法。

```dart
// error_codes.dart 扩展

abstract class AgentErrorCodes {
  // ... 现有错误码 ...

  /// 获取用户友好的错误提示
  static String getUserFriendlyMessage(String code) {
    return switch (code) {
      // JSON 管道 (E1xx)
      jsonExtractFailed ||
      jsonSanitizeFailed ||
      jsonSchemaInvalid ||
      jsonStructuralRepairFailed ||
      jsonLlmRepairFailed => 'AI 正在整理思路，请稍候...',
      jsonRepairFailed => 'AI 无法生成有效格式，建议简化输入后重试',

      // Agent 执行 (E2xx)
      agentExecutionError => 'AI 遇到意外错误',
      agentTimeout => 'AI 响应超时，请检查网络后重试',
      agentUnknown || agentNotRegistered => '功能暂不可用',
      agentInvalidInput => '输入内容无效，请检查后重试',

      // 输出处理 (E3xx)
      outputTruncated => 'AI 结果过长，已自动精简',
      outputSummarizeFailed => 'AI 摘要生成失败',
      outputTooLarge => 'AI 结果超出限制',

      // Worker (E4xx)
      workerCrash => '后台服务正在重启...',
      workerTimeout => '处理超时，请稍后重试',
      workerCommunicationError => '内部通信错误',

      // LLM (E5xx)
      llmCallFailed => 'AI 服务暂时不可用',
      llmInvalidResponse => 'AI 返回了无效响应',
      llmTimeout => 'AI 响应超时',

      _ => '发生未知错误: $code',
    };
  }

  /// 判断是否应自动重试（不打扰用户）
  static bool shouldAutoRetry(String code) {
    return switch (code) {
      jsonExtractFailed ||
      jsonSanitizeFailed ||
      jsonSchemaInvalid ||
      jsonStructuralRepairFailed ||
      llmTimeout => true,
      _ => false,
    };
  }

  /// 判断是否需要用户介入
  static bool requiresUserAction(String code) {
    return switch (code) {
      jsonRepairFailed ||
      agentInvalidInput ||
      outputTooLarge => true,
      _ => false,
    };
  }
}
```

### 2.5 PERF-04: JSON 管道优化

**问题**: 6 阶段管道可能叠加多次 LLM 调用。

**方案**: 根据模型可靠性跳过阶段 + 限制 LLM Repair 次数。

```dart
// model_adapter.dart 扩展

class ModelAdapter {
  // ... 现有代码 ...

  /// 判断模型 JSON 输出是否可靠
  ///
  /// 可靠模型跳过 S3/S4 阶段
  bool isReliableJson(String modelId) {
    final normalized = modelId.toLowerCase();

    // GPT-4, Claude 3+ 等大模型 JSON 可靠
    if (normalized.contains('gpt-4') ||
        normalized.contains('claude-3') ||
        normalized.contains('claude-opus') ||
        normalized.contains('claude-sonnet')) {
      return true;
    }

    // 小模型或未知模型需要完整管道
    return false;
  }
}
```

### 2.7 UX-02: 版本冲突处理

**问题**: 版本闸门拒绝 Proposal 时用户无感知。

**方案**: Auto-Retry on Stale + 失败时 Toast 提示。

```dart
// rp_version_gate.dart 扩展

class RpVersionGate {
  /// 验证 Proposal 版本
  ///
  /// 返回 (isValid, shouldAutoRetry)
  (bool, bool) validate(RpProposal proposal, RpVersionInfo current) {
    if (proposal.expectedFoundationRev != current.foundationRev) {
      // Foundation 变更，无法自动重试
      return (false, false);
    }

    if (proposal.expectedStoryRev != current.storyRev) {
      // Story 变更，可尝试自动重试
      final drift = current.storyRev - proposal.expectedStoryRev;
      if (drift <= 2) {
        // 小幅漂移，自动重试
        return (false, true);
      }
      // 大幅漂移，需用户确认
      return (false, false);
    }

    return (true, false);
  }
}
```

## 6. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 超时值过短误杀慢模型 | 功能失效 | 分模型配置超时，大模型给更长时间 |
| 取消信号丢失 | 资源浪费 | 超时作为兜底，双重保险 |
| 进度消息过多 | 性能损耗 | 节流发送，100ms 内合并 |
| 预热占用资源 | 内存增加 | 仅在 Roleplay 页面预热 |
