# 写作/跑团式角色扮演：需求基底（Draft）

> 目标：在不牺牲沉浸感的前提下，系统性解决 LLM 在“长对话/互动小说”场景中的记忆与一致性问题。
>
> 本文用于固化需求基底，减少后续频繁重构；后续实现细节另行文档化（不在本文做落地映射）。
>
> 最后更新：2026-01-07

---

## 0. 范围与定位

### 0.1 产品形态

- 形态：AI 驱动的互动小说 / 跑团式角色扮演（用户作为故事内角色参与叙事推进）。
- 重点：记忆、上下文编排、角色一致性、剧情世界线、token 成本与稳定性（而非固定“扮演分工”）。

### 0.2 本阶段的“必须解决”

- **角色一致性**：外观（发色/瞳色/体貌）与行为语言风格不跑偏。
- **剧情世界线**：关键节点不遗忘、状态不自相矛盾。
- **可迭代设定**：关键节点出现后，设定/事实能及时更新并可追溯。
- **token 可控**：世界书/角色卡/多辅助 agent 不应导致单次 tokens 失控；长对话退化要可控。
- **透明可溯源**：用户可不看，但系统必须记录完整日志与版本历史。

## 1. 已确认的需求决策（当前共识）

### 1.0 用户输入（In‑character vs OOC）

- ✅ 已确认：用户以“角色身份”输入通常是为了**引导剧情推进**（有明确指向/意图），系统应按“正常用户输入”处理（参与剧情、可触发事件/状态/目标等维护）。
- 备注：这不意味着“自动写入 Canon”。任何写入 Canon 的事实仍需人类在环确认（见 1.2/1.3）。

### 1.1 场景 vs 时间线

- 以 **场景（Scene）为主索引**，同时保留时间线能力（Scene 内可能发生事件序列）。

### 1.1.1 Scene 的默认切换方式

- 默认采用：**系统提议切场景 → 用户确认**（避免误切导致沉浸感受损）。

### 1.2 事实写入原则（Canon）

- **仅写入明确事实**到“硬记忆/Canon”。
- 推测/不确定信息不得直接写入 Canon（如需记录，必须走“待确认/待澄清”渠道，而不是混入硬记忆）。

### 1.2.1 Canon 的默认确认方式（写入门槛）

- 默认采用：**AI 提议写入 → 用户确认**（允许/拒绝/修改后允许）。
- 用户也可以主动发起“写入/确认为 Canon”的操作，但最终仍应落到同一套“提议与确认”流程中，以保证可审计与可回滚。

### 1.3 记忆更新的人类在环（Human-in-the-loop）

- 采用 **AI 提议 → 弹窗说明理由与证据 → 展示更新内容（可编辑）→ 用户允许/拒绝** 的流程。
- 回滚：至少支持 **一键撤销本次更新**；更细粒度（按条目）作为可选增强。

### 1.4 记忆形态（块 → 条目）

- 记忆以 **Memory Block（大类）→ Entry（条目）** 的层次组织：
  - Block 例：事件、人物关系、物品/状态、地点/世界观规则、伏笔等。
  - Entry：对该类的结构化或半结构化描述（条目化便于版本化与追溯）。

### 1.4.1 块内视图（Block → View → Entry）

- 各块可提供 **View（局部视图/子块）**：对同一块内条目的筛选/分组/聚合呈现（不一定引入新数据源）。
- 用途：UI 组织与注入选择（例如“到期伏笔”“当前目标”“某角色的物品清单”“本 Scene 相关事件”）。

### 1.5 用户可操作的核心资产

- 角色卡
- 世界观（世界书/世界百科）
- 记忆条目（可查看、可编辑、可回滚）
- 文风与叙事规则（可编辑）

### 1.6 角色卡版本策略（默认）

- 采用 **基准角色卡 + 剧情修订层（Overlay/Delta）** 作为默认策略：
  - 剧情推进导致的变更（例如外观变化、关系变化、状态变化）优先落在修订层，便于追溯与回滚。
  - 是否“提升为基准”（写回基准角色卡）由用户决定。

### 1.6.1 角色记忆（主观/认知）

- 需要区分“客观事实（世界/事件）”与“角色主观认知（角色相信什么/误解什么/情绪印象）”。
- ✅ 已确认：允许记录“角色的错误信念/误解”，以提升真实感与视角一致性；但 Canon 写入仍遵循“仅写入明确事实”原则（例如：可以写入“角色认为 X”，而不是直接写入“X 为真”）。

### 1.6.2 Threads 拆分（目标 vs 伏笔）

- ✅ 已确认：将 Threads 拆分为两个块：**Goals/Tasks** 与 **Foreshadow/Hooks**（后续可提供聚合视图用于统一查看）。

### 1.6.3 关键事件（Key Events）与伏笔进度

- ✅ 已确认：**关键事件由系统判定为主**（自动抽取/评分），用户可手动编辑/标记“是否关键”。
- ✅ 已确认：伏笔需要“进度/状态条（Tracker）”，并同时覆盖两类信号：
  - **证据积累程度**（与伏笔相关的关键事件/证据链在增长）
  - **回收接近程度**（故事推进到“适合回收/即将回收”的阶段）
  - 两者按权重综合（权重可配置；默认由系统决定，用户可调整）
- ✅ 已确认：关键事件本身是“全局高价值事件”（不要求与伏笔相关）；伏笔需要一层“关联器（Linker）”从关键事件中挑出与该伏笔相关的事件子集，并允许记录 `positive/negative/neutral` 及对两条轴的影响量（用户可手动调整影响量/影响率）。
  - ✅ 补充：允许存在 `negative`，但默认 Linker 不自动产出 negative；negative 关联默认需要用户审查确认（避免误判），并支持后续用“reinterpretation 关联”而非翻转历史记录来表达“真相大白后的再解读”。

### 1.6.3 Mechanics（可选）

- ✅ 已确认：允许加入检定/骰子/规则模块；不一定高频使用，但应可启用。

### 1.7 日志与透明度（最低要求）

- 必须可追溯（默认可隐藏），并至少覆盖：
  - **token 统计**（每轮、分层/分块预算与实际，或至少输入/输出 token）
  - **对各个块/条目的更新记录**（提议内容、用户编辑、最终写入、回滚点）
  - **注入内容清单**（本轮注入了哪些内容，来自哪里，为什么）

### 1.8 多端同步（目标）

- 目标：实现 **全同步**（手机端与电脑端一致）。
- 当前：**暂不实现多端同步**（成本/服务器原因）。
- 备注：为降低未来迁移成本，需求层应提前约束数据与版本模型（例如稳定 ID、变更日志、可合并语义），避免将来“全量重构存储层”。

### 1.9 可拆卸与可拓展（模块化）

- 需求：块级能力应可像 Mod 一样 **启用/禁用**，并遵循统一框架与规则约束，便于拆卸与拓展。
- ✅ 已确认：**每个模块/卡片都是非必要的**（可不启用；不启用时系统仍能正常对话）。
- ✅ 已确认：**禁用=停止注入与后台维护，数据保留**；禁用后 Orchestrator 与后台系统应完全忽略该模块。
- ✅ 已确认：禁用后 **禁止编辑**，需再次启用后才允许编辑。
- ✅ 已确认：模块间依赖如存在，采用 **软依赖（suggests）+ 降级路径**；不使用硬依赖（requires）。
- ✅ 已确认：导出格式暂定 **仅 JSON**（先不引入 zip/分片格式）。

### 1.10 编辑模式（权限与沉浸）

- ✅ 已确认：提供编辑模式以控制“可编辑范围与 UI 暴露”，避免沉浸被频繁打断。
- ✅ 已确认：默认对用户暴露 **Standard + Immersive**；`God/Editor` 作为高级开关。
- ✅ 已确认：沉浸模式下允许编辑 **Style**（不改变事实，只改变叙事体验）。

### 1.11 Style（文风/叙事约束）是否存在的语义

- ✅ 已确认：当 **未配置 Style 卡片** 时，系统不强行约束叙事视角/文风（由 LLM 根据上下文自行决定）。
- ✅ 已确认：当 **配置了 Style 卡片** 时，系统应按 Style 要求注入并约束输出（这正是 Style 的意义）。

### 1.12 Proposal（提议）权限分级

- ✅ 已确认：Proposal 支持按类型/模块/模式配置的三档权限分级：
  - 静默更新（Silent）
  - 提醒后直接更新（Notify‑Apply）
  - 提醒且要求用户审查确定（Review‑Required；Canon‑Write 默认必须）
- ✅ 已确认：冲突通常不阻塞输出处理；更重要的是把用户的纠正行为（如编辑对话内容的修改）转为可追踪信号，交给专用 agent 产出相应修订提议。

## 2. 关键设计点（需要明确但尚未定稿）

> 以下不是实现方案，而是“必须在需求层确认”的设计点；确认后才适合进入工程设计。

### 2.1 角色卡的“基准版”与“剧情修订”

你提出的疑问：剧情推进中角色设定会变化，是否需要“二级角色卡”？

建议在需求层明确三种模式之一（择一为主，其他可后续扩展）：

- **模式 A：直接修改角色卡**（简单，但历史追溯与“自动写错”风险更高）
- **模式 B：基准角色卡 + 剧情修订层（Overlay/Delta）**（更推荐：自动更新先落在修订层，可回滚；用户可“提升为基准”）
- **模式 C：分层角色卡**（外观/性格为稳定层，状态/关系为运行层；更复杂但表达清晰）

需要你最终拍板：哪一种是“默认模式”。

> ✅ 已确认：默认采用“模式 B：基准角色卡 + 剧情修订层（Overlay/Delta）”。

### 2.2 伏笔（Foreshadow）机制的最小表达

你倾向：系统自动抽取为主、用户可手动创建；伏笔频率较低。

需要明确：

- 伏笔的最小字段：例如 `content`、`strength`、`introduced_at(scene/turn)`、`expected_payoff`（可选）、`status`（待回收/已回收）等。
- “回收”的判定：由 AI 自动提议？还是用户显式标记？还是混合？

### 2.3 世界观注入方式：关键词 vs 语义检索

你目前无偏好：希望“最优最灵活”。

建议需求层先给出优先级：

- **优先 1：关键词触发（可控、便宜、可解释）**
- **优先 2：语义检索（召回强，但更费 token/成本且需更强去噪与预算）**

✅ 已确认：**并存，但默认只开关键词触发；语义检索作为可选增强**。

### 2.4 分支/消息树与 active chain

如果对话存在“消息树/分支线程”（类似你项目中的消息树结构），需求层需要明确：

- 默认上下文、摘要、检索、记忆更新：**只基于当前 active chain**（避免串线）。
- 是否允许跨分支检索（例如“全局搜索”，但不自动注入，仅供用户/工具查看）。

### 2.5 分支情况下的“记忆如何处理”（关键）

你提出的核心疑问：分支剧情过长会导致记忆发生较大变化，切回原路线时记忆怎么办？

需求层建议先明确两层概念（避免后续推翻重构）：

- **基底（Base/Foundation）**：初始角色卡/世界书/Style 等“基本不变的底座”；跨分支共享；不随剧情回滚。
- **Canon‑Story（随剧情演进的已确认事实）**：只在“用户确认写入”后才生效；按分支/active chain 隔离；应随剧情回滚同步回滚。
- **Draft/Working（待确认变更）**：可随分支变化；切换分支时应跟随分支（或被丢弃/合并，取决于最终策略）。

需要你决定：Draft 是否要“分支隔离”，以及切换分支时 Draft 如何迁移/清空。

✅ 已确认（更新表述以避免歧义）：**基底跨分支共享；`canon_story` 与 Draft 按分支隔离**（切换分支时跟随分支；剧情回滚需同步回滚 `canon_story`）。

### 2.6 背景维护（后台处理）与沉浸感

你允许后台维护，但希望不破坏沉浸感。需求层需明确：

- 后台维护的触发点：每轮？每 N 轮？场景结束？仅手动？
- 后台维护的可见性：对用户完全静默？还是仅在“日志/状态”可见？

### 2.7 纠错 vs 剧情变化（冲突处理）

当新内容与 Canon 冲突（如发色、身份、关系）时，系统需要区分：

- **纠错**：之前记错了（模型/人工错误），应覆盖旧 Canon，并记录纠错原因。
- **剧情变化**：设定在剧情中发生改变，应新增“变化事件/当前状态”，而不是静默覆盖旧 Canon。

建议默认：**冲突一律进入提议弹窗，由用户选择“纠错/剧情变化/拒绝写入”**，避免系统擅自改 Canon。

### 2.8 多 Agent / 多 Prompt 体系（形态与边界）

你提出希望以多个 agent（记忆整理/总结/概念构建等）来实现系统能力，并采用“指挥官/调度”模式：

- 只有“指挥官（Orchestrator）”常驻运行；它决定何时启动哪个专用 agent/任务。
- 专用 agent 使用对应 prompt + 工具完成单一任务（如：记忆抽取、冲突检测、场景压缩、伏笔整理等）。
- 不是所有 agent 每轮都工作；默认应由指挥官依据触发条件与预算来调度，避免 token 与延迟失控。

需求层需要先明确：

- “多 agent”是产品层的显式角色（用户可见、可切换），还是系统内部的任务管线（用户不可见）？
- 后台任务的触发频率与预算上限（避免 token 失控）。
- 关键原则：**任何写入 Canon 的变更都必须走 1.3 的人类在环流程**（不因 agent 多而绕过确认）。

## 3. 观测与审计（必须有，但可默认隐藏）

### 3.1 日志分类（建议）

- Context 编译日志：本轮注入了哪些块/条目、各层预算与实际消耗、裁剪/降级原因。
- 记忆维护日志：触发了哪些提议、提议理由、用户操作（允许/拒绝/修改）、最终写入结果。
- 检索日志：检索查询、召回结果摘要、过滤原因、最终被注入的片段（如有）。
- 版本历史：每次记忆更新的 diff、回滚点、恢复操作。

### 3.2 最小可追溯字段（建议）

- 触发来源（用户消息/系统规则/后台 agent）
- 时间与会话/场景标识
- 影响范围（哪些 blocks/entries 被改）
- 理由/证据（引用到的对话片段或条目）

## 4. 下一轮需要你回答的问题（用于锁定“稳固基底”）

> 你可以直接用编号回答；不确定也可以选“暂不定”，我们会把它标为开放项。

1) “明确事实”的判定：对你来说，“明确事实”的来源边界是什么？例如：用户口头陈述是否默认可信？是否必须点“确认为设定/写入 Canon”才算？
2) Scene 提议切换的触发信号：你希望系统基于哪些信号提议切场景？（地点变化/时间跳转/目标完成/用户显式指令/其他）
3) 伏笔的最小字段：你更希望伏笔条目包含哪些字段？（content、status、introduced_at、expected_payoff、trigger、tags…）
4) 冲突弹窗：当提议更新与 Canon 冲突时，弹窗里你希望默认提供哪些选项？（纠错/剧情变化/拒绝/手动编辑后写入/推迟处理）
5) 多 Agent 的“内外边界”：专用 agent 是完全内部（用户只看到结果与日志），还是用户能手动启动/查看中间过程？
6) 日志导出：是否需要一键导出（用于复现/反馈 bug）？需要脱敏吗？
7) 性能底线：你能接受一次“记忆提议弹窗”前的等待上限大概是多少？（例如 <1s / <3s / <10s）
