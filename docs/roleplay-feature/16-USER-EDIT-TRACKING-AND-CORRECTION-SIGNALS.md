# 用户对话编辑追踪与纠错信号（Draft）

> 目标：把“用户手动纠正对话内容”的行为变成系统可利用的强信号：可追踪、可审计、可回滚，并能驱动专用 agent 产出更精准的记忆/设定更新 Proposal。
>
> 你提出的典型例子：用户把“蓝发”编辑为“黑发”，系统应能追踪这次修改，并把它交给专用 agent 去更新相应块级内容。
>
> 本文不做工程落地映射，只描述语义、边界与流程。
>
> 最后更新：2026-01-07

---

## 0. 为什么“编辑对话”是高价值信号

相比模型自己总结/推测，用户直接改字往往意味着：

- 这是你最在意的细节（否则不会去改）
- 这是明确的纠错或设定修订（尤其是外观、状态、地点、时间）
- 这是可用于修正长期记忆的“强监督信号”

因此建议把“用户编辑”视为一种特殊事件：**Correction Signal**。

---

## 1. 追踪什么（Edit Event / Revision）

当用户编辑某条消息（用户消息或 AI 消息）时，系统应记录一个 Edit Event（或 Message Revision）：

- `message_id`（被编辑的消息）
- `branch_id / active_chain_id`（作用域：避免跨分支污染）
- `editor`：`user`（将来也可扩展为 `system`）
- `edited_at`
- `before_text` / `after_text`
- `diff`（结构化差异；至少能定位到“改了哪些片段”）
- `edit_reason`（可选）：让用户选一个原因能大幅降低误判：
  - `typo/format`（纯排版/错别字）
  - `style`（润色，不改事实）
  - `fact_correction`（事实纠错）
  - `retcon`（剧情改写/设定改动）
  - `safety`（删改敏感内容）

> 你未要求用户每次都选 reason：可默认不选，由 agent 分类；但建议在 UI 中提供“可选的原因”按钮，能显著提高系统稳定性。

## 2. 编辑后的处理流程（概念）

### 2.1 Step A：噪声过滤（避免把润色当纠错）

系统可先做一些轻量过滤（不依赖 LLM）：

- 只改标点/空格/换行 → 归类为 `typo/format`
- 变化很小且不含实体（人名/地点/颜色/数量/时间）→ 低优先级
- 变化涉及实体或数值（发色、物品数量、地点名、时间）→ 高优先级

### 2.2 Step B：交给专用 agent 做“分类 + 指向”

专用 agent 的输入：
- `diff`
- 当前 Scene（若启用）
- 相关 RoleCard/World/State/Timeline 的 Quick Views（按预算）

输出（作为 Proposal 或队列项）：
1) 这次编辑更像：
   - 纠错（修正之前的错误）
   - 剧情变化（设定/状态在剧情中发生改变）
   - 视角差异（角色误解/撒谎/信息不对称）
   - 纯文案润色（不应改记忆）
2) 应该更新哪个模块/块/条目：
   - `character`（外观/性格/口癖/Delta）
   - `state`（物品/伤势/关系 flags）
   - `timeline`（关键事件纠错或补充）
   - `world`（世界设定纠错）
   - `foreshadow_hooks`（修正关联/进度信号）
3) 产出具体 diff（可回滚、可审计）与证据链（优先引用 Edit Event 本身）

### 2.3 Step C：生成 Proposal（走权限分级）

你已确认 Proposal 有三档权限：
- `silent`
- `notify_apply`
- `review_required`

对“用户事实纠错”类建议默认 `review_required`（强信号，但仍可能是 retcon 或润色）。

---

## 3. 与“模块禁用”的关系（你要求的“彻底忽略”）

你已确认：模块禁用后系统应全方面忽略且禁止编辑。

因此这里建议：
- Edit Event **仍然会被记录**（因为它属于对话历史的一部分）
- 但专用 agent **只对已启用模块生成 Proposal**
  - 如果目标模块被禁用：只写日志（“因模块禁用未处理”），不生成 Proposal

---

## 4. 示例（对齐你的直觉）

### 示例 1：外观纠错

用户把 AI 回复中的 “蓝发” 改成 “黑发”：
- agent 分类：`fact_correction`（或 `retcon`，取决于上下文）
- 建议更新：`character` 的 Quick Facts 或 Delta（发色字段）
- 证据：Edit Event diff + 引用最近对话片段

### 示例 2：状态纠错

用户把 “我把钥匙放回口袋” 改成 “我把钥匙交给了守卫”：
- 建议更新：`state`（物品归属变更），可能同时更新 `timeline`（关键事件）

### 示例 3：视角差异

用户把 “NPC承认了” 改成 “NPC否认了”：
- 建议：更新 `timeline`（事件描述纠错）+ `foreshadow_hooks`（可能新增 negative/neutral 关联，但默认需审查）

## 5. 你可能还要决定的 3 个小开关

1) ✅ 是否允许在 Immersive 模式下编辑历史消息？（你已确认：允许，并且默认显示入口；原因：模型可能抽风，需要随时快速纠错）
2) 编辑会不会影响“已发送上下文”的摘要/关键事件？（建议：会影响，但必须走 Proposal，避免静默篡改历史）
3) 是否允许把 Edit Event 也作为 Key Event 候选？（例如用户大幅改写了一段剧情，这可能就是一个关键事件/设定变更点）
