# 分支（消息树）+ 回滚 + 后台任务：一致性与竞态处理（Draft）

> 目标：在存在消息树/分支、支持回滚、并允许后台维护（异步 agent）时，给出一套“准确且高效”的语义约束，避免：
> - 回滚后被旧后台任务写回去
> - 切分支后把 A 分支的 Draft 污染到 B 分支
> - 编辑旧消息后摘要/关键事件/伏笔关联失真
>
> 本文不做工程落地映射（不讨论 Flutter/Hive/线程实现），只定义“数据语义与安全闸门”。
>
> 最后更新：2026-01-07

---

## 0. 你已确认的基础规则（作为前提）

- **基底（Base/Foundation）跨分支共享；随剧情演进的数据按分支隔离**：
  - `canon_story`（已确认、但随剧情变化的事实）按分支隔离，且应随回滚同步回滚
  - `draft`（推测/临时/待确认）按分支隔离
- 模块禁用后：不注入、不维护、不可编辑、后台完全忽略
- 回滚必须存在（至少一键撤销本次更新）

本文件在上述规则上补齐“竞态与一致性”。

---

## 1. 推荐的“版本化语义”（核心：让后台任务可安全丢弃）

### 1.1 为什么必须版本化

只要有后台任务，就会出现这种情况：
1) agent A 基于旧上下文算出提议/关联/摘要
2) 用户切分支/回滚/编辑旧消息
3) agent A 结果晚到，覆盖了新状态

解决思路：**任何后台产物都必须绑定“它基于哪个版本的世界”**，否则结果一律视为过期。

### 1.2 版本号（概念）

建议至少维护以下 revision（可以是整数递增或 hash）：

- `foundation_rev`：基底版本（跨分支共享；例如初始角色卡/世界书基底/写作 Style 等）
- `canon_story_rev(branch)`：某分支“随剧情演进的已确认事实”版本（分支隔离；回滚应同步回滚）
- `draft_rev(branch)`：某分支 Draft 版本（分支隔离）
- `source_rev(branch)`：源对话版本（该分支 active chain 的消息序列/编辑历史版本）

> 直觉：后台任务的输入来自“对话源 + 当前记忆状态”。任一变动都应让旧任务失效。

---

## 2. 后台任务的“安全闸门”（Job Safety Gate）

每个后台任务在创建时必须记录：

- `job_scope`：`story_id + branch_id + module_id + task_type`
- `input_snapshot`：
  - `foundation_rev_at_start`
  - `canon_story_rev_at_start`
  - `draft_rev_at_start`
  - `source_rev_at_start`
  - 可选：`enabled_modules_hash`（避免禁用后仍执行）

在任务完成准备应用结果时，必须二次检查：

- 如果任一 `*_rev` 不匹配当前状态 → **任务过期（stale）**，结果丢弃或转为“建议重新运行”
- 如果模块已禁用 → 丢弃

> 这条规则能一刀切解决你担心的“回滚后被后台写回去”的问题。

---

## 3. 分支隔离：哪些东西跟分支走，哪些跨分支

### 3.1 必须分支隔离（默认）

- Draft 条目（包含：伏笔进度/置信度、Linker 关联、重要性评分、摘要草稿等）
- `canon_story` 条目（已确认事实，但随剧情演进，应随回滚同步回滚）
- Proposal 队列（因为它基于分支语境）
- Scene State（不同分支的当前场景可能不同）
- Key Events（Draft 级默认分支隔离；确认后写入 `canon_story`（同分支）；如需提升为“基底”，必须走显式的提升流程）

### 3.2 跨分支共享（必须谨慎）

- 基底（Base/Foundation）：初始角色卡（基底层）、世界书基底层、写作 Style 等

> 关键：Timeline 里可以同时存在 Draft Key Events 与 Canon Key Events，但它们都应以“分支/active chain”为作用域，避免把“某分支才发生的事件”硬写到其他分支。

---

## 4. 回滚的两种层级（建议同时支持）

### 4.1 回滚“某次更新”（最小必须）

- 一键撤销“本次应用的 Proposal/补丁”
- 本质是把某次 `operation` 反向应用（或把指针回到上一个 revision）

### 4.2 回滚“到某个快照”（增强但非常有用）

- 选择某个历史节点（例如某次关键事件确认/某次场景切换），一键回到那一刻的：
  - `foundation_rev`（可选：通常不建议随便回滚基底，除非 God/Editor）
  - `canon_story_rev(branch)`（随剧情演进的已确认事实，应随回滚同步回滚）
  - `draft_rev(branch)`
  - `source_rev(branch)`（对应的消息链版本）

> 你不急着做多端同步，但“快照/版本化”会让未来迁移与调试成本大幅下降。

### 4.3 “快照级回滚”在实际中的价值（用情境理解）

下面用 TRPG/互动小说的常见情况解释：为什么仅有“撤销本次更新”往往不够，而“回到某个快照”会成为非常实用的兜底能力。

**情境 A：想试探剧情，但不想把试探结果污染当前线**
- 你让用户角色尝试一个“也许很蠢”的行动（比如鲁莽挑衅强敌）只是想看看走向。
- 这会连带触发：Scene 摘要变化、Key Event 增加、伏笔进度变化、甚至角色关系 Draft 更新。
- “撤销本次更新”只能撤掉最后一个补丁；但试探可能产生的是一串更新（多条 Proposal + 多个模块的 Draft 变化）。
- 用“快照级回滚”就能一键回到试探前的状态，避免你手动逐条撤销、也避免漏撤导致后续一致性继续被污染。

**情境 B：一次后台维护产生了连锁漂移**
- 系统在某个节点做了较强的后台维护（例如：压缩前情/重建时间线/重算伏笔 Linker），但结果让人物语气、动机、或事实对照变得“怪”。
- 你可能需要快速回到维护前，对比差异并重新发起更保守的维护策略（或让用户逐条审查）。
- 快照级回滚让这种“回到维护前 + 换策略再试”变成低成本循环。

**情境 C：沉浸模式里误触/误接受导致大量 silent 更新**
- 即便你设计了权限分级，仍可能出现：某次批量 silent 更新写入了不该写入的 Draft/甚至错误 Canon（极端情况）。
- 此时用户需要“紧急刹车按钮”：一键回到某个被信任的节点（比如上一次确认后）。
- 快照级回滚比“逐条回看日志 + 逐条撤销”更符合沉浸体验。

**情境 D：编辑旧消息触发重算，但重算不可靠**
- 用户把旧消息里的某个细节改正（例如把“蓝发”改为“黑发”），系统据此触发 Key Events/Scene 摘要/一致性约束的局部重算。
- 如果重算失败或引入新错误，快照级回滚可以让用户/系统立刻回到“编辑前”，再选择更谨慎的重算窗口或更强审查级别。

**情境 E：版本升级/更换模型/改提示词**
- 你切换模型或更新提示词体系后，输出风格与维护行为可能改变。
- 快照级回滚可作为“升级前检查点”：如果新版本表现不佳，快速退回，避免在同一条故事线上造成不可逆的混乱。

> 直觉总结：分支解决“平行世界”；快照解决“同一条线上的可恢复性”。两者叠加，才能同时满足探索与稳定。

## 5. 编辑旧消息（diff 追踪）如何与分支/回滚协同

你已经提出 diff 追踪 + 专用 agent，这是正确方向。这里补齐“作用域与失效策略”：

### 5.1 编辑旧消息会导致哪些派生数据失效？

至少应标记以下派生物为 dirty（等待重算或生成 Proposal）：

- Key Events 抽取结果（timeline）
- Foreshadow Linker 关联（foreshadow_hooks）
- 场景摘要/previously_on（scene）
- 一致性闸门的“事实对照表”（character quick facts/state constraints 的引用）

### 5.2 失效范围如何控制（效率）

为“高效”而非全量重算，建议：

- 只重算“受影响的窗口”（例如：编辑消息所在位置附近的一段）
- 并且重算结果也必须受 Job Safety Gate 约束（绑定 rev）

## 6. 仍需你确认的 3 个关键选择

1) ✅ **Draft Key Events 默认分支隔离**（你已确认；确认写入 Canon 后才共享）
2) ✅ 回滚“快照级”纳入长期目标（非 MVP；先定语义/数据结构，UI/交互可后做）
3) ✅ 编辑旧消息在 Immersive 下 **默认开放且默认显示入口**（用于快速纠错/修复模型抽风造成的不合理剧情/对话）
