# 块级结构与记忆流转（Draft）

> 目标：为“写作/跑团式角色扮演”建立稳定的 **块级（板块）结构** 与 **记忆更新/注入** 的共同语言，便于后续逐步落地（本文不做落地映射）。
>
> 关键词：块→条目、Canon/Draft、Scene、低 token、可回滚、可溯源、AI 自动维护 + 人类在环。
>
> 最后更新：2026-01-07

---

## 0. 术语（对齐用）

> 你提出的“板块级结构（块级）→ 条目”是合理的；下列术语用于减少我们对同一概念的不同叫法。

- **Block（块/板块）**：用户可见的一级结构（例如“角色卡”“世界书”“伏笔”），内部包含条目。
- **Entry（条目）**：块内的最小可版本化单元；可增删改、可回滚、可引用证据。
- **Canon（硬记忆/已确认事实）**：必须可审计；只允许写入“明确事实”（来自用户陈述/明确剧情文本/用户确认写入）。
  - 为满足“剧情回滚”，建议把 Canon 再细分为两层：
    - `foundation`（基底）：基本不变的底座（初始角色卡/世界书/Style）；不随剧情回滚
    - `canon_story`（随剧情演进）：已确认事实但随剧情变化；应随剧情回滚同步回滚
- **Draft / Working（工作记忆/未确认）**：允许包含推测、临时总结、候选更新；不会被当作“事实约束”强制模型遵循。
- **Proposal（提议/变更单）**：AI 生成的“对某些块/条目的更新建议”，包含理由与证据；用户允许/拒绝/编辑后落地。
- **Scene（场景状态）**：当前“正在发生的这一段互动”的结构化快照，用于：减少漂移、做局部记忆维护、做 token 编排的锚点。
- **Active chain（当前分支链）**：消息树中当前选中的对话链；默认所有“摘要/检索/记忆维护”只基于 active chain。
- **View（局部视图/子块）**：块内的“筛选/分组/聚合视图”（不一定是独立数据源），用于 UI 展示与注入选择（例如“到期伏笔”“当前目标”“某角色的物品清单”）。

## 1. 为什么 Scene 在这里很重要（不是“章节”，而是“锚点”）

你希望系统能稳定处理“长上下文下的设定漂移”，而不是只靠“越塞越多的设定”。Scene 的价值是：

1) **为注入内容提供最强的相关性过滤器**：同一世界书里 100 条设定，当前 Scene 只需要 3 条。  
2) **为记忆维护提供触发点**：切场景/目标完成/时间跳转/新角色登场，是“该更新记忆”的天然信号。  
3) **为压缩提供边界**：按 Scene 生成“前情摘要/关键事件”，比全局摘要更稳、更可回滚。  

你当前希望 Scene 最小字段为：地点、在场角色、当前目标/冲突、时间、精简前情（OK，足够作为 v0）。

## 2. 推荐块级结构（v0，可扩展）

> 目标不是把所有东西“都注入”，而是把所有东西“都可管理”，再由指挥官（Orchestrator）按需注入。

### 2.0 总览表：块 × 数据层（帮你快速对齐“Canon 到底是什么”）

> **重要澄清**：`foundation / canon_story / draft` 是同一个块内的 **数据层级**，不是三个独立块。  
> 默认语义：**剧情回滚应回滚** `source + scene + proposals + canon_story + draft`；**不回滚** `foundation`（但 foundation 仍应有独立的撤销/审计）。

| Block（模块） | `foundation`（基底，不随剧情回滚） | `canon_story`（已确认事实，随剧情回滚） | `draft`（推测/候选，随剧情回滚） | 典型条目（示例） |
|---|---|---|---|---|
| 角色（Character） | ✅ | ✅ | ✅ | 基底角色卡；剧情修订 Delta/Overlay；外观/关系/状态变更；角色记忆（“我知道/我相信/我撒谎说”） |
| 世界书（World/Lorebook） | ✅ | ✅ | ✅ | 世界规则/地点/组织（基底）；世界状态变化（城市毁灭等）；传闻/未证实设定 |
| 场景状态（Scene State） | — | ✅ | ✅ | 当前地点/在场角色/时间/目标/精简前情；候选“切场景/目标变化”提议 |
| 时间线（Timeline/Key Events） | — | ✅ | ✅ | 已确认关键事件（发生了什么）；候选关键事件/重要性评分（待确认） |
| 目标与任务（Goals/Tasks） | — | 可选 | ✅ | “当前主目标/子目标/下一步”；默认作为系统工作台（Draft），必要时可手动确认成 `canon_story` |
| 伏笔与钩子（Foreshadow/Hooks） | — | 可选（证据/引入点） | ✅ | 伏笔证据链（“文本里确实出现过”）；关联器 Link + 进度条（证据积累/回收接近）+ 置信度等判断字段 |
| 状态与约束（State/Flags） | — | ✅ | ✅ | 物品归属/伤势/关系标记/已知情报；候选状态变更与冲突提议 |
| 机制（Mechanics） | 可选（规则基底） | 可选 | 可选 | 跑团规则/数值；掷骰/裁定记录；未确认的建议/候选检定 |
| 文风与叙事规则（Style） | ✅ | — | 可选 | 人称/时态/禁用词/节奏；（可选）风格候选提议（默认不主动打断） |

### 2.1 必备块（强烈建议）

1) **角色（Character）**
   - 内容：角色卡（基准）+ 剧情修订层（Delta/Overlay）+（可选）角色记忆/知识（Character Memory/Knowledge）。
   - 为什么：你关心的发色/瞳色/行为语言一致性，本质都在这里收敛；而“角色记忆/知识”用于解决 **视角一致性**（角色只知道自己知道的）与 **避免元信息泄露**。
   - 关键点：
     - Delta 应可逐条回滚；必要时支持“提升为基准”。
     - 角色记忆/知识不要等同于“全局事件台账”：同一事件对不同角色的“已知/未知/误解/情绪反应”可能不同。

2) **世界书（World / Lorebook）**
   - 内容：世界观规则、地点、组织、常识、禁忌、术语表等条目化知识。
   - 默认注入：关键词触发（ST World Info 思路）+ 预算/优先级裁剪。

3) **场景状态（Scene State）**
   - 内容：当前 Scene 的结构化快照（你给出的 5 项字段）。
   - 默认注入：几乎每轮都应出现，但必须极短（“一屏以内”）。

4) **时间线 / 事件台账（Timeline / Event Ledger）**
   - 内容：已确认的关键事件条目（“发生了什么”），可按 Scene/时间标记。
   - 为什么：解决“重要节点遗忘/串线”；也是伏笔回收与冲突判断的证据来源。

### 2.2 强烈建议的块（解决你点名的痛点）

5) **目标与任务（Goals / Tasks）**
   - 内容：当前主目标、子目标、未完成任务、下一步行动建议（可选）等。
   - 价值：让故事推进“有方向”，同时给 Orchestrator 一个稳定的“剧情推进锚点”。
   - 注入建议：P1 小段（只保留 1-3 条最紧迫目标），避免变成任务列表刷屏。

6) **伏笔与钩子（Foreshadow / Hooks）**
   - 内容：伏笔（埋下/到期提醒/已回收）、悬念钩子、待揭示线索等。
   - 价值：直接对应“伏笔遗忘”“重要伏笔漏回收”。
   - 关键点：把“证据（文本中确实出现过）”与“解读/预期回收方式（可能是推测）”分开存，符合“Canon 只存事实”。
   - 注入建议：默认只注入“到期/高强度”的 top-N（例如最多 2 条）。
   - 你提出的增强（建议作为 v0+）：为伏笔增加 **回收度/置信度**（进度条）
     - 进度/置信度应默认属于 Draft（可变、可回滚），并且必须带“理由 + 关联证据条目”。
     - 伏笔条目可关联多个“关键事件”条目：当新关键事件与伏笔相关时，更新进度条并把事件挂到该伏笔下（可作为证据链的一部分）。
     - 关键澄清：Timeline 的“关键事件（Key Event）”是**全局高价值事件**（不要求与伏笔相关）；伏笔模块需要一层 **关联器（Linker）** 来从 Key Events 中挑出“与该伏笔相关”的子集（伏笔关联事件）。
       - 关联关系建议支持：`positive / negative / neutral`（大多数情况是 positive，但允许 negative 表示阻碍/反证/误导；都默认 Draft，需证据链与理由）。
       - 你已确认：用户手动标记/编辑时，可为某个事件配置“对该伏笔进度的影响率/影响量”。
     - 你已确认：进度条同时包含两类信号（按权重综合）：
       - **证据积累程度**（evidence accumulation）
       - **回收接近程度**（payoff proximity）
       - 权重可配置（默认系统决定，用户可调整）

7) **状态与约束（State / Flags）**
   - 内容：物品、伤势、buff/debuff、关系状态标记、已知情报状态等（以条目/flag 形式）。
   - 价值：减少“上一轮受伤，下一轮满血”“拿到钥匙却又说没拿”的低级错误。
   - 你已确认的方向：作为 **独立块** 存在；但允许在“角色块”下提供“角色局部视图”（例如只显示该角色拥有的物品/状态）。

### 2.3 可选块（看你是否要做“更像跑团”）

8) **机制（Mechanics / Dice / Rules）**
   - 内容：骰子规则、数值、检定结果记录、GM 裁定等。
   - 如果你不强调规则系统：可以先只做 `roll` 工具与结果记录（条目化）。

9) **文风与叙事规则（Style / Narrative Rules）**
   - 内容：人称/时态、文风禁用词、节奏偏好、暴力/成人内容边界等。
   - 你已确认的方向：用户可编辑；作为 Author’s Note 类机制按频率/条件注入（避免每轮大段塞入）。

## 2.4 Threads：合并 vs 拆分（结合场景的取舍）

你已决定将 Threads 拆分为两个块（Goals / Foreshadow+Hooks）。这里保留对比，作为后续“可切换模块策略”的依据：

### 方案 A：合并为一个 Threads 块（类型化条目）

**适合场景**：
- 跑团/互动小说里，“目标”“线索”“伏笔”经常在同一段 Scene 里互相转化（目标达成→伏笔回收；线索出现→新目标生成）。
- 你希望 Orchestrator 在“沉浸不中断”的情况下，快速拿到“当前未竟事项全景”。

**优势**：
- **统一排序与预算**：同一套“紧迫度/重要性/到期程度”排序，最容易做 token 裁剪（例如只注入 top-N）。
- **天然支持转化**：Goal 完成后可以自动把条目状态改为 Resolved；Foreshadow 回收后同一条目状态迁移即可（不用跨块搬家）。
- **UI 更像任务面板**：一个“线程面板”就能看到“所有未竟事项”。

**缺点**：
- **语义更杂**：不同类型的条目字段不同（Goal 可能有 deadline/next_step；Foreshadow 可能有 planted_at/evidence）。需要更强的 schema 约束，或允许子类型扩展。
- **检索噪声风险**：当你只想召回“到期伏笔”，合并块里会混入目标/任务，需要过滤规则更清晰。

### 方案 B：拆分为两个块：Goals / Foreshadows+Hooks

**适合场景**：
- 你把“目标/任务”当作强产品心智（像任务系统），而“伏笔/钩子”更像创作辅助提示。
- 你希望 UI 与权限更清晰（例如：Goal 更允许系统自动维护；Foreshadow 更强调用户确认与证据引用）。

**优势**：
- **规则更明确**：每块字段更统一、自动维护策略更容易定（Goal：生成/推进/完成；Foreshadow：埋下/到期提醒/回收）。
- **注入更稳定**：例如 P1 永远注入当前 Goal；P2 才注入 Foreshadow（避免“伏笔提示”过于打断沉浸）。

**缺点**：
- **跨块一致性成本**：Goal 的完成经常意味着某条 Foreshadow 被回收，拆开后需要做“跨块联动/引用”。
- **预算裁剪更难全局最优**：Goal 块与 Foreshadow 块各自裁剪，可能不是整体最优（例如本轮 Foreshadow 比 Goal 更重要）。

✅ 你已选择：**默认采用方案 B（拆分）**。但这不妨碍提供一个“Threads Dashboard（聚合视图）”把两块内容汇总展示/注入（见 `10` 的 View 契约）。

## 3. 注入编排（Token 可控的关键）

块级增加不等于 token 增加——**关键是“编排器只注入当轮必要的最小集合”**。推荐借鉴 MuMu 的 RTCO/P0-P2 思想：

- **P0（必须）**：系统约束 + 当前 Scene State + 极少量“硬约束快照”（例如主角外观关键字段）。
- **P1（重要）**：本轮涉及角色的 Delta、命中的世界书条目、最紧迫的 Goals 与 Foreshadow/Hooks（例如“当前目标/到期伏笔”）。
- **P2（可选）**：回忆检索（对话历史搜索 / 语义检索）与更长的背景材料；预算不足时优雅降级。

> MuMu 的一个细节值得直接学习：把“待回收伏笔”优先于“相关记忆”注入，并且严格限制数量与长度（例如最多 2 条、每条截断）。这非常符合你要的“不能没有，但可以默认隐藏/可控”的风格。

## 4. 记忆更新流转（以你认可的“指挥官 + 专用 agent”形态描述）

### 4.1 Orchestrator 的工作方式（建议）

每轮只做两件事：
1) **决定“要不要维护/维护哪些块”**（Scene、角色 Delta、事件、伏笔、状态……）
2) **决定“本轮注入哪些内容”**（并记录注入清单与预算）

其余专用 agent（记忆抽取、冲突检测、伏笔整理、摘要压缩等）按需启动，并受预算约束。

### 4.2 提议→确认→写入（你已确认的流程）

1) 专用 agent 产出 Proposal：包含
   - 目标块/条目
   - diff（新增/修改/删除）
   - 理由（触发原因）
   - 证据（引用到的消息片段/事件条目）
2) UI 弹窗：用户 **允许/拒绝/编辑后允许/暂存待处理**
3) 写入：仅当用户允许后，才写入 Canon（或写入对应 Delta/State/Timeline）
4) 版本与回滚：至少支持“一键撤销本次更新”；理想情况支持按条目撤销

## 5. 从 research 借鉴的“可直接抄作业”的策略（不涉及落地映射）

### 5.1 Letta（最值得学的不是“有记忆”，而是“可控的记忆工程”）

- **Memory Blocks 一等公民**：每块有 `label/description/value/limit`，并有硬上限校验（避免越写越炸）。
- **编辑工具化**：以“补丁/替换/追加/完成编辑”等原子操作更新记忆，而不是整段重写。
- **系统提示词编译**：把 Memory / Tool Rules / Metadata 等拼成可复现的 system prompt。
- **工具输出截断 + 退化路径**：工具输出过长直接截断；上下文超窗有总结/裁剪 fallback。

### 5.2 SillyTavern（适合你“世界书/角色卡”担忧的 token 解法）

- **World Info 关键词触发 + Scan Depth**：只扫最近 N 条消息，只注入命中条目。
- **Token Budget + Priority**：预算不够时按优先级裁剪（而不是“全都塞进 prompt”）。
- **@Depth / Author’s Note**：把少量高权重提醒放在对话靠后位置，抵抗漂移。

### 5.3 MuMuAINovel（伏笔与动态裁剪的可复用部分）

- **伏笔状态机**：`0 普通 / 1 已埋下 / 2 已回收`，并记录强度/来源章节（在互动中可替换为 Scene/事件 ID）。
- **到期伏笔提醒（lookahead）**：当“埋下后经过 N 个单位仍未回收”，把它提升为注入优先级，并限制条数（例如最多 2 条）。
- **结构化分析→记忆条目**：把“章节/片段分析结果”转为条目（情节点、角色变化、冲突、钩子），并带重要性分数用于过滤。

### 5.4 Arboris（如果你希望“更像创作系统”，可后续再引入）

- **五层信息架构 + RAG**：蓝图（固定）/摘要（检索）/文本块（检索）/桥接（固定）/当前输入（固定）。
- **多版本生成+评审**：适合“写作”场景，但对“沉浸式跑团”可能要谨慎使用（会打断节奏）。

## 6. 已确认的边界（汇总）

- Threads：拆分为 **Goals/Tasks** 与 **Foreshadow/Hooks** 两块，但允许提供聚合视图（Threads Dashboard）。
- 关键事件（Key Events）：系统判定为主，用户可手动编辑/标记；并作为证据源与“时间推进单位”。
- 伏笔到期单位：**N 个关键事件**（从“埋下”起累计发生的关键事件数；不要求事件与该伏笔相关）。
- 伏笔进度条：双轴（证据积累 / 回收接近）+ 权重聚合；默认 Draft；必须可回滚、可解释、可引用证据链。
- 伏笔关联器（Linker）：从 Key Events 中挑出“与该伏笔相关”的事件，并记录 `positive/negative/neutral` 以及对两条轴的增减；支持用户手动调整“影响量/影响率”。
- State/Flags：独立块 + 角色局部视图。
- Style：用户可编辑，且可完全不启用（不启用时系统仍可正常对话）。
- Mechanics：可选模块（低频也可存在）。
- 模块禁用语义：禁用=停止注入与后台维护+禁止编辑；Orchestrator 与后台系统应完全忽略该模块。

> 为了实现你说的“统一框架 + 可拆卸/可拓展”，建议所有块都遵循同一份模块契约（manifest/schema/budget/triggers/log/view）。详见：`docs/roleplay-feature/10-BLOCK-MODULE-FRAMEWORK.md`。
