### FOLDER 11: lib/data - ModelCapabilityPresets

**File**: model_capability_presets.dart (107 lines)

**Preliminary Assessment**:
- Pure data presets: 26 model entries (OpenAI, Claude, Gemini, DeepSeek, misc)
- Matching strategy: exact → prefix → contains (three-tier fallback)
- No complex logic, but W-001 to W-006 issues identified

**Key Issues to Verify**:
1. **Matching Rule Duplication** (W-001): getCapabilities() and isKnownModel() both implement matching logic separately. Should extract to private helper _matchesModelPattern().
2. **Missing Cache** (W-002): getCapabilities() performs O(n) scan on _presets.keys each call. High-frequency queries (dialog rendering) will be slow.
3. **Set Allocation** (W-003): Each getCapabilities() call returns `Set.from(entry.value)` - creates new Set instead of reusing const. Wasteful for immutable presets.
4. **String.substring() Edge Case** (W-006, line 104): getSuggestedDisplayName() does `word[0].toUpperCase() + word.substring(1)` - empty string word causes IndexOutOfBoundsException.
5. **Language Detection Coverage**: Does this preset cover all 50+ LLM models in market (Claude Haiku/Opus/Sonnet/Nova, GPT-4o/mini, Gemini 1.5-pro/flash/exp, DeepSeek-v3, etc)? Missing entries will fall back to "text-only".

**Codex Task**: Detailed audit
- Confirm string edge case vulnerability
- Assess caching impact: how often getCapabilities() called?
- Verify model coverage completeness vs latest LLM releases
- Check if new models should have dynamic loading support (config file vs hardcoded)

### FOLDER 12: lib/design_system - Design Tokens

**File**: design_tokens.dart (223 lines)

**Preliminary Assessment**:
- Pure design token definitions: spacing, radius, elevation, animation, breakpoints
- Excellent structure and documentation
- Minor issues: W-001 (no multi-theme support), W-003 (appIcon() boundary)

**Key Issues to Verify**:
1. **Light/Dark Theme Support** (W-001): Current design uses const tokens only. How are Light/Dark theme colors handled? (Likely in separate OwuiTokens class, but confirm separation of concerns).
2. **appIcon() Boundary** (W-003, line 101): `BorderRadius.circular(appIcon(0))` passes size=0 → appIcon(0) = 0*0.23 = 0 → circular(0) OK, but appIcon(-10) = -2.3 → circular(-2.3) error? Validate input constraint.
3. **elevation Shadows vs Material Elevation**: Custom shadow definitions override Material3 elevation system. Is this intentional? Could break when Material updates elevation specs in future releases.
4. **Animation Durations**: 150ms (fast), 210ms (Apple standard), 300ms (slow), 560ms (menu), 1200ms (breathe) - is 1200ms breathe animation too slow for real-time thinking bubble? Should test UX.
5. **Breakpoint Values**: mobile=600, tablet=1024, desktop=1440 - are these aligned with Flutter ResponsiveWidget conventions? Check if widgets use these correctly.

**Codex Task**: Design system completeness audit
- Confirm color token handling strategy for Light/Dark
- Validate elevation shadow values vs Material3 spec
- Check animation duration appropriateness (especially breathing 1200ms)
- Verify breakpoint usage consistency across pages/widgets
- Suggest improvements for multi-brand/white-label support (if future requirement)
