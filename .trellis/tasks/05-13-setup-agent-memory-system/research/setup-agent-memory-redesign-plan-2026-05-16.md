# SetupAgent Memory Redesign Plan

Date: 2026-05-16

## 1. Confirmed Direction

SetupAgent memory is a session-scoped fact recovery system. Its job is to help the agent recover setup facts that were already written into the current setup workspace but are no longer fully present in the prompt because the discussion is long, the draft has many entries, or context compaction has happened.

It is not a Nocturne-style personality memory. Nocturne is valuable as a reference for active recall surfaces, but its first-person identity, emotion, and `disclosure` model should not be copied into SetupAgent. SetupAgent needs missing-fact recall, closer to a code agent opening indexed project files.

It is also not a runtime semantic classifier. The current memory-related `SetupActionExpectation` / `ActionDecisionPolicy` / completion guard chain is the wrong direction because it makes runtime code guess whether the user needs an exact fact from natural-language markers. That creates brittle policy code and shifts agent cognition into backend guard logic.

The corrected direction is:

```text
deterministic draft/truth parsing
  -> session fact index
  -> agent-visible folder-like index view
  -> setup.memory.search when ref is unknown
  -> setup.memory.open when ref is known
  -> level-3 entry refs open to level-4 section indexes
  -> level-4 section refs open to clean content
```

The agent should know that an index entry or summary is not a full fact source. When it needs established setup details that are missing from the current context, it must open the corresponding ref instead of inventing details.

## 2. Lessons From Claude Code And Nocturne

Claude Code validates the file-index direction:

- use a manifest / index instead of stuffing all content into the prompt;
- expose small searchable references first;
- read the full source only when needed;
- keep current project/workspace truth authoritative;
- avoid turning memory into a generic long-term vector RAG layer.

Nocturne validates active recall surfaces, but not its domain model:

- useful: boot/index/recent/glossary style surfaces make recall paths visible to the model;
- useful: memory tools distinguish locating a node from reading a node;
- not useful for SetupAgent: personality continuity, first-person memory authorship, emotion/persona recall, and model-authored `disclosure` conditions.

For SetupAgent, every draft entry has the same recall rule:

```text
If the agent needs an established setup fact and the current context does not contain the full detail, recall it through the index/tools.
```

There should be no per-entry model-authored `when_to_recall` field.

## 3. Current Implementation Snapshot

The current implementation already has useful foundations:

- `backend/rp/setup_agent_memory/` is modular.
- `SetupSessionMemoryService` builds a manifest from live setup sources.
- `setup.memory.search` returns candidate refs without full payloads.
- `setup.memory.read_refs` reads current DB-backed source payloads, but it is now treated as a compatibility/internal readback path rather than the recommended agent-facing workflow.
- SetupAgent tool scope exposes setup-owned memory tools and excludes RP Memory OS tools.

The current implementation also has design debt:

- memory recall is tied into `SetupActionExpectation`;
- `ActionDecisionPolicy` uses keyword-like exact-detail markers to decide whether search/read must happen;
- completion guard can block text finalization because memory search/read was not performed;
- tool batch blocking can force a search/read sequence before other actions;
- search candidates include block, entry, and section refs; section-level hits are useful and should remain available for precise recall;
- `read_refs(full)` returns raw structured payloads with internal fields instead of a clean agent-facing content view;
- there is not yet an agent-facing `setup.memory.open` tool that maps the file-system mental model onto memory refs;
- there is no stable folder-like index view in the model context.

## 4. Confirmed Design Decisions

### 4.1 Index Ownership And Maintenance

The index is generated by deterministic backend logic, not written by the model.

The index should be a derived cache over setup sources:

- source truth remains `SetupWorkspace` editable draft and accepted setup truth;
- editable draft and accepted truth are both setup fact sources for this feature, and should be normalized into the same agent-facing folder-like index/open workflow;
- handoff and runtime compact / recovery refs belong to context assembly and compaction recovery, not to the memory index/open source set;
- the cache may be refreshed after draft write/edit/delete;
- the cache may be rebuilt from source at any time;
- full draft content is never treated as owned by the index cache;
- content reads always go back to the authoritative setup source.

This gives deterministic, backend-readable, testable index behavior while avoiding a second truth store.

### 4.2 Default Index Granularity

The agent-facing default index is a folder-like tree to level three:

```text
stage / entry_type_or_category / entry
```

Examples:

```text
世界观 / 种族 / 精灵
世界观 / 国家 / 北境王国
角色 / 主角团 / 艾琳
剧情 / 主线事件 / 王都叛乱
```

Entry sections are not shown in the default index. Sections remain inside the entry and can participate in internal search text, but the default visible and readable unit is the entry.

### 4.3 Current Stage Vs Other Stages

The default context should show a full-session index, not only the current stage. Setup discussions naturally cross stage boundaries.

Confirmed presentation rule:

- current stage: show `path + ref`;
- other stages: show `path + ref + short summary`;
- over budget: preserve `path + ref` first and trim summaries before trimming refs;
- summaries are navigation aids only, not authoritative facts for precise answers.

### 4.4 Recall Capability Model

The desired agent-facing recall model has two agent-facing tools and three underlying operations:

1. locate a third-level index entry when the agent does not already know the ref;
2. open a third-level entry ref to obtain its fourth-level section index;
3. open the selected fourth-level section ref to read clean content.

The agent may skip the locate step when the third-level index is already visible in context. Search remains useful as a fallback or shortcut when the agent cannot find the right path from the visible index.

The working mental model is:

```text
agent-visible level-3 index
  -> search level-3/level-4 refs when needed
  -> open level-3 entry ref to view level-4 section refs
  -> open level-4 section ref to view clean content
```

Agent-facing tools:

- `setup.memory.search`: locate candidate refs when the visible index is insufficient.
- `setup.memory.open`: open exactly one ref.

`setup.memory.read_refs` may remain registered or implemented for compatibility with existing tests and internal reader paths, but prompt/tool guidance should make `search + open` the main model-visible workflow.

### 4.5 Open Capability

`setup.memory.open` is the agent-facing file-system-style operation. It replaces the earlier proposed three-step recall path with `search -> open`; there is no separate directory-expansion tool in the current design.

The first slice supports one ref per call. Batch open is deferred until the single-ref chain is stable.

#### Opening a level-3 entry ref

Input:

```text
stage:<stage>:<entry_id>
```

Output contract:

- `result_type: "index"`;
- `opened_ref`;
- `opened_path`;
- deterministic `message` saying this is a level-3 entry index and the agent must open a level-4 ref to view content;
- `sections` containing clean level-4 refs, titles, paths, and optional `navigation_summary`.

The level-3 open result is an index view, not content recall. It must not return full section text.

Example:

```json
{
  "success": true,
  "result_type": "index",
  "opened_ref": "stage:world_background:elf",
  "opened_path": "世界观 / 种族 / 精灵",
  "message": "当前打开的是三级条目索引，以下是该条目下的四级目录。需要查看具体内容时，请继续 open 某个四级 ref。",
  "sections": [
    {
      "path": "世界观 / 种族 / 精灵 / 栖息地",
      "ref": "stage:world_background:elf:habitat",
      "title": "栖息地",
      "navigation_summary": "精灵主要居住地相关设定。"
    }
  ]
}
```

#### Opening a level-4 section ref

Input:

```text
stage:<stage>:<entry_id>:<section_id>
```

Output contract:

- `result_type: "content"`;
- `opened_ref`;
- `opened_path`;
- deterministic `message` saying this is a content node and can be used as fact evidence;
- `content` as a clean structured block.

The level-4 content result does not include summaries. It returns the usable fact content directly.

Example:

```json
{
  "success": true,
  "result_type": "content",
  "opened_ref": "stage:world_background:elf:habitat",
  "opened_path": "世界观 / 种族 / 精灵 / 栖息地",
  "message": "当前打开的是四级内容节点，以下内容可作为回答或写入草稿的事实依据。",
  "content": {
    "type": "text",
    "title": "栖息地",
    "text": "精灵主要生活在月光森林..."
  }
}
```

Opening a level-3 entry ref must not return the full entry content in the first slice. If the agent needs an overall answer, it should open the entry, inspect the fourth-level section index, and then open the relevant sections one at a time.

### 4.6 Search Result Unit

`setup.memory.search` may return entry-level and section-level refs.

This matches the current implementation and supports precise recall. The always-visible index should stay at level three, but active search can return level-four section refs when that is the strongest match.

For example, if the query matches the `栖息地` section under `精灵`, the returned ref may be:

```text
stage:world_background:elf:habitat
```

Search remains a way to locate an index entry or section. It is not the fact source. Search hits may include `navigation_summary` to help choose a ref, but the summary is not evidence. The agent still needs `setup.memory.open` before using exact recalled content.

### 4.7 Agent-Facing Content Shape

`setup.memory.open` should return clean content views, not raw internal payloads. Any compatibility `read_refs(full)` output shown to the model should follow the same cleaning rule.

When opening an entry ref, the result is a fourth-level section index, not full entry content. When opening a section ref, the result should include the selected section content as a clean structured block. Minimal parent context may appear in `opened_path`; avoid extra parent payload unless later tests show the agent needs it.

Preferred section shape:

```json
{
  "result_type": "content",
  "opened_ref": "stage:world_background:elf:habitat",
  "opened_path": "世界观 / 种族 / 精灵 / 栖息地",
  "content": {
    "type": "text",
    "title": "栖息地",
    "text": "主要生活在月光森林..."
  }
}
```

Clean structured block variants:

- text: `{ "type": "text", "title": "...", "text": "..." }`;
- list: `{ "type": "list", "title": "...", "items": [...] }`;
- key-value: `{ "type": "key_value", "title": "...", "values": { ... } }`.

The exact clean-view schema is allowed to evolve. It may expand or shrink based on real agent behavior. The stable rule is that any tool output or context section shown to the agent should contain useful recall content and hide internal/debug fields by default.

Do not expose these fields to the model by default unless a later product decision explicitly needs them:

- `entry_id`
- `semantic_path`
- `fingerprint`
- `freshness`
- `source_kind`
- `ref_kind`
- `block_type`
- `tags`
- `section_id`
- `kind`
- `retrieval_role`
- score/reason/debug metadata

Those may remain available in debug traces, service-level objects, or tests, but not in the main agent-facing recall payload.

## 5. Context Assembly Contract

Context assembly should provide a map and boundary awareness. It should not decide on behalf of the agent which memory must be read.

Each model request should include:

1. current setup stage / task state;
2. current working context;
3. session memory index view;
4. compact summary / recovery guidance when compaction happened, owned by the context layer rather than the memory index;
5. lightweight guidance:
   - index entries and summaries are navigation only;
   - if exact established setup facts are needed and the current prompt lacks full detail, use the visible index or `setup.memory.search` to locate the ref, then use `setup.memory.open`;
   - opening a level-3 ref returns a level-4 directory, not fact content;
   - opening a level-4 ref returns clean fact content;
   - if opened content is still insufficient, continue `search/open` related refs or ask the user; do not infer missing facts from genre/type/common sense.

The context layer should not prefetch full entry content by default. Prefetch can be considered later only for highly deterministic cases, such as a user directly naming a unique path/ref, but it is not part of the first cleanup slice.

## 6. Cleanup Slice

The first implementation slice should remove the current hard runtime chain and replace it with prompt/context/tool guidance.

Planned changes:

- remove memory-specific `search_session_memory` / `read_memory_refs` expectations from `SetupActionExpectation`;
- remove memory keyword classifiers from `ActionDecisionPolicy`;
- remove memory-specific tool batch blocking;
- remove completion guard behavior that blocks finalization only because memory search/read was not called;
- remove `action_expectation` from runtime overlay or reduce it to non-memory deterministic repair state;
- update prompt/tool guidance so the agent uses the index and memory tools when it lacks established facts;
- update tests that currently assert forced search -> read -> answer.

Runtime guards should remain for deterministic failures only, such as invalid tool output, tool repair obligations, repeated no-progress questions, and commit/readiness rules.

## 7. Index Capability Slice

After cleanup, implement the index-facing improvements:

- add an agent-facing session index view built from deterministic backend parsing;
- represent the index as a folder-like tree grouped by stage and entry type/category;
- expose current stage as `path + ref`;
- expose other stages as `path + ref + short summary`;
- keep search capable of returning section-level hits;
- add `setup.memory.open` so a third-level entry ref returns fourth-level section refs;
- make a fourth-level `setup.memory.open` result return clean structured section content;
- keep any compatibility `read_refs(full)` output aligned with the same clean-view rule;
- keep debug metadata out of the model-facing payload but available for traces/tests.

## 8. Validation Plan

Focused tests should cover:

- memory-related runtime expectation/guard code no longer forces search/read;
- `setup.memory.search` and `setup.memory.open` are the recommended agent-facing memory tools; `setup.memory.read_refs` remains compatible where still required;
- index view is deterministic and grouped as `stage / category / entry`;
- current stage index rows omit summary while other-stage rows include bounded summary;
- search can return section refs when section content is the strongest match;
- open of a level-3 entry ref returns fourth-level section refs without full section content;
- open of a level-4 section ref returns clean structured content without internal structural/debug fields;
- compact context exposes recovery guidance without raw historical discussion; those recovery artifacts are context-layer inputs, not memory index/open sources;
- real or near-real model probe checks whether the model voluntarily recalls through tools when only index/summary is visible.

## 9. Open Questions

- Whether the derived index cache needs a table in the first index slice or can initially be rebuilt and attached as context on demand.
- Accepted truth and editable draft refs are merged into the same agent-facing folder tree as setup fact sources. Any source distinction is internal/debug metadata, not a separate recall workflow.
- Whether legacy refs such as `foundation:<entry_id>` should be normalized into the same folder path view or only preserved as compatibility refs.
