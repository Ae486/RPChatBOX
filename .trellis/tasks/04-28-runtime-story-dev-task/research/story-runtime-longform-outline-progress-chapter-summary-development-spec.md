## 2. Required Decisions

The following decisions are frozen for implementation:

1. Outline structure uses structured JSON, not Markdown splitting by `\n`.
2. One accepted segment covers one beat.
3. Chapter summary is generated by a provider that may reuse writer/model
   execution with a summary prompt.
4. Manual beat status editing is out of scope for Phase T.
5. `complete_chapter` remains a deterministic product action; any LLM summary
   call is provider-internal bridge maintenance, not a user-visible writer turn.

## 5. Backend Contract Details

### 5.1 Structured Outline DTO

Use Pydantic models for backend validation. Suggested DTOs:

```python
class LongformOutlineBeat(BaseModel):
    beat_id: str
    order: int
    title: str
    goal: str
    must_include: list[str] = Field(default_factory=list)
    avoid: list[str] = Field(default_factory=list)
    continuity_notes: list[str] = Field(default_factory=list)
    metadata_json: dict[str, Any] = Field(default_factory=dict)


class LongformStructuredOutline(BaseModel):
    schema_version: Literal["longform_outline_v1"]
    chapter_index: int
    chapter_title: str | None = None
    chapter_goal: str
    beats: list[LongformOutlineBeat]
    constraints: dict[str, Any] = Field(default_factory=dict)
```

The exact names may change to fit the existing module, but the schema must be
typed and validated.

### 5.2 Outline Materialization

`generate_outline` should produce structured outline data. If the current writer
transport cannot reliably return JSON in one step, use a narrow normalization
adapter:

1. ask writer for structured JSON when possible;
2. validate with Pydantic;
3. if validation fails, run one deterministic/LLM repair pass or fail closed;
4. persist both display text and structured outline metadata;
5. never use raw Markdown line breaks as canonical beat ids.

### 5.3 Outline Progress Record

Suggested model:

```python
class LongformOutlineProgress(BaseModel):
    outline_artifact_id: str
    chapter_index: int
    current_beat_id: str | None
    covered_beat_ids: list[str] = Field(default_factory=list)
    segment_by_beat_id: dict[str, str] = Field(default_factory=dict)
    status_by_beat_id: dict[str, str] = Field(default_factory=dict)
    metadata_json: dict[str, Any] = Field(default_factory=dict)
```

Storage can be:

1. chapter workspace metadata if it is already branch-scoped and rollback-safe;
2. Runtime Workspace sidecar if that better matches existing material lifecycle;
3. existing chapter bridge/progress record if present.

The implement agent must inspect current code before choosing. The rule is that
progress must be branch-scoped, traceable, and not Core truth.

### 5.4 Packet Assembly

`write_next_segment` must include an `outline_progress` packet section.

Minimum section items:

- `current_beat_id`
- `current_beat_order`
- `current_beat_title`
- `current_beat_goal`
- `current_beat_must_include`
- `current_beat_avoid`
- `covered_beat_ids`
- `latest_accepted_segment_excerpt`
- `instruction: write only this beat and stop before later beats`

This section must be visible in prompt rendering and inspect evidence.

### 5.5 Segment Adoption

When a segment candidate is produced, persist `target_beat_id` in artifact
metadata.

When `accept_and_continue` adopts that candidate:

1. verify the target beat is the current pending beat;
2. mark the beat accepted;
3. bind beat id to adopted segment artifact id;
4. advance `current_beat_id` to the next pending beat;
5. if there is no next pending beat, mark chapter ready for completion.

Rewrite and selection must not advance progress.

### 5.6 Chapter Summary Provider

Add or extend provider boundary:

```python
class ChapterSummaryProvider:
    def build_chapter_summary(...) -> ChapterSummaryMaterial: ...
```

If the existing code already owns chapter bridge sidecars through
`ChapterBridgeProvider`, prefer extending that provider or adding a thin summary
adapter behind it. Do not introduce a second chapter bridge truth surface.

Implementation guidance:

1. Prefer reusing existing `StoryLlmGateway` / writer execution transport rather
   than adding a new LLM stack.
2. Keep prompt assembly inside the provider, not inside the chapter domain
   transaction.
3. Require structured summary output, for example:
   - `summary_text`
   - `covered_beats`
   - `continuity_notes`
   - `open_threads`
   - `source_refs`
4. Persist the result as Runtime Workspace / chapter bridge sidecar with full
   identity and branch.
5. Do not write Recall/Core directly in Phase T.

Compatibility note:

- The earlier chapter/review adapter spec allowed a lightweight bridge provider
  and said not to invent a heavy compact summary unless a later provider
  required it.
- Phase T is that later provider requirement. It should produce a structured
  bridge summary sidecar, but it still must remain bounded: adopted segments and
  covered beats in, structured bridge material out, no Core/Recall write and no
  new framework.

## 6. Tests Required

Backend tests must focus on implemented behavior:

1. `accept_outline` creates structured beat progress from valid outline JSON.
2. invalid outline JSON does not silently become untracked Markdown progress.
3. `write_next_segment` packet contains exactly one current beat section.
4. generated segment candidate metadata includes `target_beat_id`.
5. `accept_and_continue` advances the cursor only after adoption.
6. rewrite / candidate selection does not advance the cursor.
7. after accepting beat 1, the next write packet targets beat 2 and includes
   latest accepted prose.
8. `complete_chapter` calls the chapter summary provider with adopted segments
   and covered beats.
9. next chapter packet reads current branch chapter bridge summary.
10. branch/rollback visibility prevents sibling or hidden future beat progress
   from entering the active packet.

Frontend checks, only if frontend is touched:

1. focused `flutter analyze` for touched files;
2. current beat display handles missing fields without crash;
3. no UI copy claims manual beat editing or paragraph rewrite product support.

## 7. Wrong vs Correct

### Wrong

The writer returns:

```markdown
1. Opening conflict
2. Taki chooses to lie
3. Escape scene
```

and backend splits by newline to invent beat ids.

### Correct

Writer returns or is normalized into:

```json
{
  "schema_version": "longform_outline_v1",
  "chapter_index": 1,
  "chapter_goal": "Force Taki to choose between safety and truth.",
  "beats": [
    {
      "beat_id": "beat_001",
      "order": 1,
      "title": "Opening conflict",
      "goal": "Establish immediate danger."
    }
  ]
}
```

Display Markdown can still be rendered from this structure.

### Wrong

`write_next_segment` receives only an outline digest and writes any part of the
chapter.

### Correct

`write_next_segment` receives `current_beat_id=beat_002`, beat constraints, the
latest accepted segment excerpt, and an instruction to write only that beat.
