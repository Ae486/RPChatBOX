## 1. Current Decision

Context Engineering is a cross-runtime common capability.

Setup agent should implement the shared abstraction first. Story runtime should
only consume it through adapters and policy contracts.

Reason:

- setup already has the closest working implementation;
- story runtime currently needs the capability for brainstorm / chapter bridge /
  writer packet budget, but should not create a second common engine;
- compact / summary must stay a pre-model context operation, not a memory truth
  mutation path.

## 4. Setup Agent Implementation Recommendation

Do not rewrite setup context governance from scratch.

Extract carefully around the existing setup path:

1. keep `SetupContextBuilder` as setup truth/context-packet builder;
2. keep `SetupContextGovernorService` as setup adapter/policy initially;
3. extract reusable mechanics from `SetupContextCompactionService` only where
   duplication would otherwise appear;
4. keep existing setup schema `SetupContextCompactSummary`;
5. add a common operation interface behind setup-specific wrappers;
6. preserve existing setup tests before wiring story runtime consumers.

Recommended common kernel capabilities:

- source fingerprint;
- recent-window retention helper;
- token/character budget helper;
- compact prompt invocation wrapper;
- Pydantic structured output validation;
- forbidden-field validation;
- deterministic fallback;
- usage / trace report.

## 5. Story Runtime Future Consumer Work

After setup owns the common kernel, story runtime should add adapters for:

- writer packet older-context compaction;
- writer brainstorm summary;
- chapter bridge summary;
- chapter/session review summary if needed.

V4 writer brainstorm must keep this lifecycle:

1. user enters brainstorm mode / discussion;
2. brainstorm sees writer-visible context plus user brainstorm prompt;
3. ordinary discussion does not create items or write memory;
4. user may close the session as no-op and return to writing;
5. explicit user action asks for a one-shot `brainstorm_summarize` pass;
6. summary operation returns typed `BrainstormItem` list;
7. user edits / rejects / confirms items;
8. confirmed items go to scheduler / dispatcher;
9. Core-classified items go to Core workers for governed memory changes through
   the shared Core mutation path according to worker permission;
10. non-Core wishes return review/redirect material instead of being dispatched
    as Recall or Archival brainstorm edits.

Brainstorm must not know Memory OS layer details.
It must not be used as a Recall editor or Archival editor; Archival changes go
through Story Evolution.

Worker output should stay minimal:

- required: `source_item_id`, `target_ref`, `base_revision`, `operation`,
  `field_path`, `new_value`;
- backend fills `old_value` from `base_revision + field_path`;
- `reason` and full `source_refs` are optional and should not be generated by
  default.
