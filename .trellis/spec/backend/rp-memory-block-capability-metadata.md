# RP Memory Block Capability Metadata

## Scenario: Block views expose machine-checkable mutation and history capability

### 1. Scope / Trigger

- Trigger: the memory layer already exposes Core State, Runtime Workspace, and retrieval-backed Block-compatible views, but callers cannot inspect one Block and know whether mutation, proposal visibility, or history is supported.
- Applies to:
  - `RpBlockReadService` Core State and Runtime Workspace Block envelopes;
  - `RetrievalBlockAdapterService` Recall / Archival Block-compatible views;
  - `StoryBlockMutationService` mutation guards;
  - focused backend service and API tests.
- This slice is memory-layer-only. It does not require upstream story runtime metadata producers.

### 2. Signatures

Read envelope metadata fields:

Every Block-compatible view emitted by current memory read services must include these metadata fields:

```python
{
    "read_only": bool,
    "mutation_mode": str,
    "history_mode": str,
    "proposal_visibility": str,
}
```

Mutation guard:

```python
class StoryBlockMutationService:
    def _ensure_block_mutable(block: RpBlockView) -> None: ...
```

Expected values:

| Layer / source | read_only | mutation_mode | history_mode | proposal_visibility |
|---|---:|---|---|---|
| Core State authoritative | `False` | `governed_proposal_apply` | `supported` | `supported` |
| Core State projection | `True` | `unsupported_projection_read_side` | `supported` | `empty` |
| Runtime Workspace | `True` | `unsupported_runtime_workspace_scratch` | `unsupported` | `empty` |
| Recall / Archival retrieval view | `True` | `unsupported_retrieval_backed` | `retrieval_backed` | `unsupported` |

### 3. Contracts

- Capability metadata belongs to the Block read envelope. It must be generated by the adapter/read service, not trusted from stored payload metadata.
- Generated capability metadata must override conflicting stored metadata such as `read_only=False` on retrieval hits or projection rows.
- Authoritative Core State blocks remain the only mutable Block family in this slice, and mutation still goes through the existing proposal/apply workflow.
- Projection, Runtime Workspace, Recall, and Archival Block views are read-only. They must not gain a direct mutation backend.
- `StoryBlockMutationService` must reject `metadata["read_only"] is True` before attempting to normalize operations into proposal persistence.
- The existing layer guard must remain after the metadata guard so malformed or legacy Blocks cannot bypass authoritative-only mutation by omitting metadata.
- This slice must not create `rp_blocks`, replace retrieval-core physical storage, replace `WritingPacketBuilder`, or widen the public memory tool family.

### 4. Validation & Error Matrix

| Condition | Expected behavior |
|---|---|
| Authoritative Core State Block | `read_only=False`, governed mutation metadata, proposal submission still works through proposal/apply |
| Projection Block | `read_only=True`, proposal list remains `[]`, mutation/detail/apply routes reject with `memory_block_mutation_unsupported` |
| Runtime Workspace Block | `read_only=True`, proposal list remains `[]`, history/provenance remain unsupported |
| Retrieval-backed Block view | `read_only=True`, retrieval metadata survives, capability fields override conflicting hit metadata |
| Block metadata says read-only | `StoryBlockMutationService` raises `MemoryBlockMutationUnsupportedError` |
| Block metadata missing but layer is non-authoritative | `StoryBlockMutationService` still raises `MemoryBlockMutationUnsupportedError` via the layer guard |

### 5. Good/Base/Bad Cases

- Good: a formal authoritative Core State row with conflicting stored `read_only=True` still emits `read_only=False` and `mutation_mode="governed_proposal_apply"` because generated envelope capability metadata wins.
- Good: a retrieval hit with conflicting `mutation_mode` still emits `mutation_mode="unsupported_retrieval_backed"` while preserving unrelated retrieval metadata.
- Base: a legacy non-authoritative Block without capability metadata is still rejected by the existing layer guard.
- Bad: trusting stored row metadata to decide whether a projection is mutable.
- Bad: allowing Runtime Workspace draft/discussion Blocks to enter proposal normalization because they are database-backed rows.
- Bad: adding a new public memory tool or `rp_blocks` table to express capability metadata.

### 6. Tests Required

- Core State read-side tests assert authoritative/projection capability metadata.
- Runtime Workspace read-side tests assert read-only capability metadata.
- API `/memory/blocks` serialization keeps the capability fields visible.
- Retrieval adapter tests assert Recall / Archival Block-compatible views are read-only and do not mutate input hit metadata.
- Mutation tests assert read-only Block views are rejected with the existing stable unsupported error code.

### 7. Wrong vs Correct

#### Wrong

```python
metadata = deepcopy(row.metadata_json or {})
```

This trusts stored metadata to define write capability, so one stale or conflicting row can make a projection look mutable.

#### Correct

```python
metadata = {
    **deepcopy(row.metadata_json or {}),
    "route": "core_state_store",
    "read_only": True,
    "mutation_mode": "unsupported_projection_read_side",
    "history_mode": "supported",
    "proposal_visibility": "empty",
}
```

The envelope owns capability metadata and overrides conflicting stored metadata.
